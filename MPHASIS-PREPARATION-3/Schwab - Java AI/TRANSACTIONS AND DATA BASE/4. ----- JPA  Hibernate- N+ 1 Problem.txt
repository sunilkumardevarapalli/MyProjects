

JPA/Hibernate – N+1 Problem
***************************

	The N+1 Query Problem is one of the most common performance issues in JPA/Hibernate.

	It occurs when Hibernate executes:

		1 query to fetch parent entities
		N additional queries to fetch related child entities

	Result: N+1 total database queries, causing significant performance degradation.

Example:
********

@Entity
public class Department {

    @OneToMany(mappedBy = "department")
    private List<Employee> employees;
}

@Entity
public class Employee {

    @ManyToOne
    private Department department;
}

Step 1: Fetch Departments
*************************

	List<Department> departments =departmentRepository.findAll();

	SELECT * FROM department; --> Suppose 3 departments are returned.

Step 2: Access Employees
************************

	for (Department dept : departments) {
		System.out.println(dept.getEmployees().size());
	}
	
	1 query -> Departments
	3 queries -> Employees

	Total = 4 queries
	
	This is the N+1 Problem.
	
Why It Occurs

By default:

	@OneToMany
	private List<Employee> employees;

	is typically loaded LAZY.
	
Solutions:

1. Fetch Join (Recommended): Use JPQL JOIN FETCH.	

		@Query("""
			   SELECT d
			   FROM Department d
			   JOIN FETCH d.employees
			   """)
		List<Department> findAllWithEmployees();
		
2. Entity Graph

	@EntityGraph(attributePaths = "employees")
	@Query("SELECT d FROM Department d")
	List<Department> findAllDepartments();

3. Batch Fetching

	Configure Hibernate: spring.jpa.properties.hibernate.default_batch_fetch_size=20

	@BatchSize(size = 20)
	@OneToMany(mappedBy = "department")
	private List<Employee> employees;

	SELECT * FROM employee
	WHERE department_id IN (1,2,3);
	
4. DTO Projection: Avoids loading entire object graphs.

	When only specific data is needed:

	@Query("""
		   SELECT new com.example.DepartmentDTO(
			   d.id,
			   d.name
		   )
		   FROM Department d
		   """)
	List<DepartmentDTO> findDepartmentSummary();	
	
5. Avoid Unnecessary EAGER Fetching

	Many developers try:@OneToMany(fetch = FetchType.EAGER)
	
	This is usually not recommended because:

		Loads data even when not required.
		Can create larger queries and memory overhead.
		May introduce different performance issues.

		Prefer: 		FetchType.LAZY
	
with:

Fetch Joins, 
EntityGraph, 
Batch Fetching, or 
DTO Projections.


Real-World Example:

	List<Order> orders = orderRepository.findAll();

	for (Order order : orders) {
		System.out.println(order.getCustomer().getName());
	}

		1 Query -> Orders
		100 Queries -> Customers

		Total = 101 Queries


	@Query("""
		   SELECT o
		   FROM Order o
		   JOIN FETCH o.customer
		   """)
	List<Order> findAllWithCustomer();

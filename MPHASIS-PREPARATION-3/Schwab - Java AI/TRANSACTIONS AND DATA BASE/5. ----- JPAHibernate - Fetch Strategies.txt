

JPA/Hibernate Fetch Strategies

	Fetch Strategy defines when and how related entities are loaded from the database.

	Proper fetch strategy selection is critical for:

		Performance optimization
		Avoiding unnecessary database queries
		Preventing N+1 query problems
		Reducing memory consumption
		
1. EAGER Fetching

		@Entity
		public class Employee {

			@Id
			private Long id;

			@OneToOne(fetch = FetchType.EAGER)
			private Address address;
		}

		Employee employee = entityManager.find(Employee.class, 1L);

		SELECT e.*, a.*
		FROM employee e
		LEFT JOIN address a
		ON e.address_id = a.id
		WHERE e.id = 1;

		Advantages
			No LazyInitializationException.
			Related objects are immediately available.
		Disadvantages
			Loads unnecessary data.
			Can cause performance issues for large object graphs.
			Increases memory usage.
			
2. LAZY Fetching

		Related entities are loaded only when accessed.

		@Entity
		public class Employee {

			@Id
			private Long id;

			@OneToMany(mappedBy = "employee",
					   fetch = FetchType.LAZY)
			private List<Project> projects;
		}

		Employee emp = entityManager.find(Employee.class, 1L);

		SELECT * FROM employee WHERE id = 1;

		emp.getProjects().size();

		SELECT *
		FROM project
		WHERE employee_id = 1;

		Advantages
			Better performance.
			Loads only required data.
			Reduces memory footprint.
		Disadvantages
			May cause LazyInitializationException.
			Can lead to N+1 problems if not managed properly.
		Best Practice

			✅ Default to LAZY whenever possible.
			
Rule of thumb:
 ➡️ Keep associations LAZY by default.
 ➡️ Fetch explicitly using JOIN FETCH or EntityGraph based on the use case. This provides the 
 best balance between performance and maintainability.

JPA/Hibernate Fetch Strategies

Fetch Strategy defines when and how related entities are loaded from the database.

Proper fetch strategy selection is critical for:

Performance optimization
Avoiding unnecessary database queries
Preventing N+1 query problems
Reducing memory consumption
1. EAGER Fetching

Related entities are loaded immediately along with the parent entity.

Example
@Entity
public class Employee {

    @Id
    private Long id;

    @OneToOne(fetch = FetchType.EAGER)
    private Address address;
}

Query
Employee employee = entityManager.find(Employee.class, 1L);


Hibernate may generate:

SELECT e.*, a.*
FROM employee e
LEFT JOIN address a
ON e.address_id = a.id
WHERE e.id = 1;

Advantages
No LazyInitializationException.
Related objects are immediately available.
Disadvantages
Loads unnecessary data.
Can cause performance issues for large object graphs.
Increases memory usage.
Best Use Cases

✅ Small reference data

Employee -> Department
Order -> Status

2. LAZY Fetching

Related entities are loaded only when accessed.

Example
@Entity
public class Employee {

    @Id
    private Long id;

    @OneToMany(mappedBy = "employee",
               fetch = FetchType.LAZY)
    private List<Project> projects;
}

Query
Employee emp = entityManager.find(Employee.class, 1L);


Generated SQL:

SELECT * FROM employee WHERE id = 1;


Projects are not loaded yet.

When accessed:

emp.getProjects().size();


Hibernate executes:

SELECT *
FROM project
WHERE employee_id = 1;

Advantages
Better performance.
Loads only required data.
Reduces memory footprint.
Disadvantages
May cause LazyInitializationException.
Can lead to N+1 problems if not managed properly.
Best Practice

✅ Default to LAZY whenever possible.

Default Fetch Types in JPA
Relationship	Default Fetch Type@OneToOne	EAGER
@ManyToOne	EAGER
@OneToMany	LAZY
@ManyToMany	LAZY

Example:

@ManyToOne
private Department department;


By default:

@ManyToOne(fetch = FetchType.EAGER)

Common Fetching Techniques in Hibernate
1. Join Fetch

Loads associations using a SQL JOIN.

JPQL
SELECT e
FROM Employee e
JOIN FETCH e.projects
WHERE e.id = :id


Generated SQL:

SELECT e.*, p.*
FROM employee e
JOIN project p
ON e.id = p.employee_id
WHERE e.id=?

Benefits
Single query.
Avoids N+1 issue.
Very efficient.

Preferred approach in most applications.

2. Batch Fetching

Instead of loading entities one by one, Hibernate loads them in batches.

Configuration
@BatchSize(size = 10)
private List<Project> projects;


Suppose 100 employees are loaded.

Without batching:

1 query Employee
100 queries Project


With batching:

1 query Employee
10 queries Project


Much better performance.

3. Subselect Fetching

Loads collections using a subquery.

@Fetch(FetchMode.SUBSELECT)
private List<Project> projects;


SQL:

SELECT *
FROM project
WHERE employee_id IN (
    SELECT id FROM employee
);


Useful when loading many parent entities.

4. Entity Graph

Allows dynamic fetch plans.

Define
@NamedEntityGraph(
    name = "Employee.projects",
    attributeNodes = {
        @NamedAttributeNode("projects")
    }
)

Use
EntityGraph<?> graph =
    entityManager.getEntityGraph("Employee.projects");

Map<String, Object> hints = new HashMap<>();

hints.put("javax.persistence.fetchgraph", graph);

Employee emp =
    entityManager.find(Employee.class, 1L, hints);

Benefits
No query changes.
Dynamic fetching.
Cleaner code.
N+1 Query Problem
Problem

Load all employees:

List<Employee> employees =
    employeeRepository.findAll();


Hibernate executes:

SELECT * FROM employee;


Then:

for(Employee e : employees){
    e.getProjects().size();
}


Hibernate executes:

SELECT * FROM project WHERE employee_id=1;
SELECT * FROM project WHERE employee_id=2;
SELECT * FROM project WHERE employee_id=3;
...


Total:

1 + N Queries


This is called N+1 Query Problem.

Solution

Use JOIN FETCH.

SELECT DISTINCT e
FROM Employee e
LEFT JOIN FETCH e.projects


Only one query is executed.

LazyInitializationException

Occurs when a lazy-loaded entity is accessed outside the Hibernate session.

Example
@Transactional
public Employee getEmployee() {
    return repository.findById(1L).get();
}


Later:

employee.getProjects().size();


Error:

LazyInitializationException:
could not initialize proxy

Solutions

✅ Fetch data inside transaction

employee.getProjects().size();


✅ Use JOIN FETCH

SELECT e
FROM Employee e
JOIN FETCH e.projects


✅ Use Entity Graph

@EntityGraph(attributePaths = "projects")

Hibernate-Specific Fetch Modes
@Fetch(FetchMode.SELECT)


Loads association with separate SELECTs.

@Fetch(FetchMode.JOIN)


Loads using JOIN.

@Fetch(FetchMode.SUBSELECT)


Loads using subquery.

Example:

@OneToMany(mappedBy = "employee")
@Fetch(FetchMode.SUBSELECT)
private List<Project> projects;

Better performance	May fetch unnecessary data

Q2: What is N+1 Query Problem?

One query loads parent records and N additional queries load child records.

Q3: How do you solve the N+1 problem?
JOIN FETCH

Q4: Which fetch strategy should be default?

✅ LAZY should be the default choice. Fetch required associations explicitly using:

JOIN FETCH
@EntityGraph
DTO projections
Recommended Production Approach

Then use:

@Query("""
SELECT e
FROM Employee e
JOIN FETCH e.projects
WHERE e.id = :id
""")


Rule of thumb:
 ➡️ Keep associations LAZY by default.
 ➡️ Fetch explicitly using JOIN FETCH or EntityGraph based on the use case. This provides the best balance between performance and maintainability.

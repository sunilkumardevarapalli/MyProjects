Outbox Pattern

	publish events without losing data


	Database update + Event publish = guaranteed consistency

		🔴 The Core Problem (Dual Write Issue)

		Typical code:

			orderRepository.save(order);          // DB write
			kafkaTemplate.send("order-events");   // Event publish
			❌ What can go wrong?
				DB success ✅, Kafka fails ❌ → event lost
				Kafka success ✅, DB fails ❌ → ghost event

	👉 This is called Dual Write Problem
	
		✅ Outbox Pattern Solution

		👉 Instead of sending event directly to Kafka:

			Save event inside DB (outbox table)
			Publish later using a separate process
		
🧠 How It Works

	Step 1: Business Transaction
	--------------------------------
	Order Service:
	  - Save Order
	  - Save Event → OUTBOX TABLE
	  (Single DB transaction ✅)

	Step 2: Background Publisher
	--------------------------------
	Poll outbox table
	→ Publish to Kafka
	→ Mark as published
		
	Save Order + Event (Atomic)
	@Transactional
	public void createOrder(Order order) {

		orderRepository.save(order);

		OutboxEvent event = new OutboxEvent();
		event.setAggregateId(order.getId());
		event.setEventType("ORDER_CREATED");
		event.setPayload(toJson(order));
		event.setStatus("NEW");

		outboxRepository.save(event);
	}

	👉 Important: Same transaction → no inconsistency	
	
	Outbox Publisher (Poller)

		@Scheduled(fixedDelay = 5000)
		public void publishOutboxEvents() {

			List<OutboxEvent> events = outboxRepository.findByStatus("NEW");

			for (OutboxEvent event : events) {
				try {
					kafkaTemplate.send("order-events", event.getPayload());

					event.setStatus("PUBLISHED");

				} catch (Exception e) {
					event.setStatus("FAILED");
				}

				outboxRepository.save(event);
			}
		}


	🔥 Production-Level Enhancements

		✅ 1. Idempotency (VERY IMPORTANT)

			Consumer should handle duplicate events:

			if (alreadyProcessed(eventId)) return;

		✅ 2. Retry Mechanism
			Retry failed events
			Exponential backoff
			
		✅ 3. Dead Letter Queue (DLQ)

		If event fails multiple times → send to DLQ

		✅ 4. Event Ordering

			Use:

			aggregateId as Kafka key
			Ensures ordering per entity
			
		✅ 5. Cleanup Strategy

			Outbox table grows fast:

			Delete old records
			Archive to cold storage
			
“Outbox Pattern solves the dual-write problem by storing events in the same database transaction as business data and publishing them asynchronously, ensuring reliability and eventual consistency in distributed systems.”

## ✅ Step 2: Save Order + Outbox (Single Transaction)

		```java
		@Service
		public class OrderService {

			@Autowired
			private OrderRepository orderRepo;

			@Autowired
			private OutboxRepository outboxRepo;

			@Transactional
			public void createOrder() {

				Order order = new Order();
				order.setId(UUID.randomUUID().toString());
				order.setStatus("CREATED");

				orderRepo.save(order);

				// Create event
				OutboxEvent event = new OutboxEvent();
				event.setId(UUID.randomUUID().toString());
				event.setEventType("OrderCreated");
				event.setPayload(convertToJson(order));
				event.setStatus("NEW");
				event.setCreatedAt(LocalDateTime.now());

				outboxRepo.save(event);
			}
		}
		```

		👉 💥 Both saved OR both rollback

## ✅ Step 3: Background Publisher (Scheduler)

```java
@Component
public class OutboxPublisher {

    @Autowired
    private OutboxRepository repo;

    @Autowired
    private KafkaTemplate<String, String> kafka;

    @Scheduled(fixedDelay = 5000)
    public void publishEvents() {

        List<OutboxEvent> events = repo.findByStatus("NEW");

        for (OutboxEvent event : events) {
            try {
                kafka.send("order-events", event.getPayload());

                event.setStatus("SENT");
                repo.save(event);

            } catch (Exception e) {
                // retry later
            }
        }
    }
}
```

# 🔥 7. Why Not Use @Transactional + Kafka?

Because:

* DB uses **2-phase commit? ❌**
* Kafka uses separate transaction ❌
* Distributed transaction = heavy + slow

👉 Outbox = simple + reliable

# 🚨 10. Common Mistakes

❌ Publishing event directly after DB save
❌ Not marking event as SENT
❌ No retry mechanism
❌ No idempotency handling
❌ Large payload (store minimal data)

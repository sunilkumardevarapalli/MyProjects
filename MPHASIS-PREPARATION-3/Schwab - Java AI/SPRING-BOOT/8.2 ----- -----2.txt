# 1. How to configure Virtual Threads in Spring Boot

You need **Java 21+**. In current Spring Boot, virtual threads can be enabled with one property. ([Home][1])

## application.properties

```properties
spring.threads.virtual.enabled=true
```

That's it.

Or YAML:

```yaml
spring:
  threads:
    virtual:
      enabled: true
```

Official reference: [Spring Boot virtual-thread configuration](https://docs.spring.io/spring-boot/reference/features/spring-application.html?utm_source=chatgpt.com)

# 3. How many Virtual Threads per Carrier Thread?

## Important answer

> **There is no fixed number like 10 VT per Carrier or 100 VT per Carrier.**

❌ Wrong:

```text
Carrier-1 = 100 Virtual Threads
Carrier-2 = 100 Virtual Threads
```

There is **no fixed mapping**.

Instead:

```text
Many Virtual Threads
        |
        v
Small pool of Carrier Threads
```

Example:

```text
1,000 Virtual Threads
        |
        v
8 Carrier Threads
```

Conceptually:

```text
VT-1    VT-2    VT-3    VT-4
VT-5    VT-6    VT-7    VT-8
VT-9    VT-10   ...
...
VT-1000
          |
          v

--------------------------------
Carrier Thread Pool
--------------------------------
Carrier-1
Carrier-2
Carrier-3
Carrier-4
Carrier-5
Carrier-6
Carrier-7
Carrier-8
```

Many virtual threads share a much smaller set of platform/carrier threads. ([OpenJDK][3])

# 4. How many Carrier Threads are created?

By default, the JDK virtual-thread scheduler's target parallelism is based on the number of **available processors**. In the Java 21 reference implementation, `jdk.virtualThreadScheduler.parallelism` defaults to the number of available processors. ([Oracle Docs][4])

Suppose:

```text
CPU = 8 available processors
```

Conceptually, default scheduler parallelism is:

```text
8 Carrier Threads
```

Architecture:

```text
100,000 Virtual Threads
           |
           v
-----------------------------------
Virtual Thread Scheduler
-----------------------------------
           |
           v
8 Carrier Threads
-----------------------------------
Carrier-1
Carrier-2
Carrier-3
Carrier-4
Carrier-5
Carrier-6
Carrier-7
Carrier-8
           |
           v
CPU
```

### Easy formula

```text
Default target carrier parallelism
        ≈
Number of available processors
```

For example:

| Available processors | Default target parallelism |
| -------------------- | -------------------------: |
| 2                    |                          2 |
| 4                    |                          4 |
| 8                    |                          8 |
| 16                   |                         16 |

**Important:** think of this as scheduler **target parallelism**, not a permanent guarantee that exactly N carrier thread objects always exist at every instant. The JDK scheduler manages the carrier pool. ([Oracle Docs][5])

# 5. Example: 10 Virtual Threads and 2 Carrier Threads

Assume:

```text
CPU available processors = 2
```

We have:

```text
VT-1
VT-2
VT-3
VT-4
VT-5
VT-6
VT-7
VT-8
VT-9
VT-10
```

Carrier threads:

```text
Carrier-1
Carrier-2
```

Initially:

```text
Carrier-1 ---> VT-1
Carrier-2 ---> VT-2
```

Diagram:

```text
VT-1 ---> Carrier-1 ---> OS Thread-1 ---> CPU
VT-2 ---> Carrier-2 ---> OS Thread-2 ---> CPU

VT-3 ---> waiting to run
VT-4 ---> waiting to run
VT-5 ---> waiting to run
VT-6 ---> waiting to run
VT-7 ---> waiting to run
VT-8 ---> waiting to run
VT-9 ---> waiting to run
VT-10 --> waiting to run
```

Now VT-1 executes:

```java
Thread.sleep(5000);
```

Conceptually, the virtual thread can unmount:

```text
VT-1
 |
 v
WAITING
```

Carrier-1 becomes available:

```text
Carrier-1 ---> FREE
```

Scheduler executes VT-3:

```text
Carrier-1 ---> VT-3
```

Current state:

```text
Carrier-1 ---> VT-3
Carrier-2 ---> VT-2

VT-1 ---> WAITING
```

Then VT-2 blocks:

```text
VT-2 ---> WAITING
```

Carrier-2 can execute VT-4:

```text
Carrier-1 ---> VT-3
Carrier-2 ---> VT-4
```

This mount/unmount scheduling model is the key reason a small carrier pool can support very large numbers of virtual threads. ([OpenJDK][2])

# 6. So how can 8 Carrier Threads handle 1 lakh Virtual Threads?

This is the important concept.

Suppose:

```text
100,000 Virtual Threads
```

And:

```text
8 Carrier Threads
```

At **one instant**, roughly only the virtual threads mounted on available carriers are actually executing Java code:

```text
Carrier-1 ---> VT-100
Carrier-2 ---> VT-205
Carrier-3 ---> VT-10
Carrier-4 ---> VT-5
Carrier-5 ---> VT-2000
Carrier-6 ---> VT-700
Carrier-7 ---> VT-900
Carrier-8 ---> VT-50
```

Others may be:

```text
WAITING for Database
WAITING for REST API
WAITING for Kafka
WAITING for File I/O
WAITING for sleep/timer
RUNNABLE and waiting for scheduler execution
```

Example:

```text
100,000 Virtual Threads

90,000 ---> Waiting for DB / API / I/O

9,992  ---> Runnable / waiting for execution

8      ---> Currently mounted and executing
```

The exact numbers change continuously, but this is the conceptual model.

# 8. Can we configure Carrier Thread count?

At the JDK scheduler level, yes. Java 21 documents these scheduler system properties: `jdk.virtualThreadScheduler.parallelism` and `jdk.virtualThreadScheduler.maxPoolSize`. The default parallelism is the number of available processors, and Java 21 documents a default maximum pool size of 256. ([Oracle Docs][4])

For example:

```bash
java \
-Djdk.virtualThreadScheduler.parallelism=16 \
-jar employee-service.jar
```

Meaning:

```text
Target Scheduler Parallelism = 16
```

Architecture:

```text
Virtual Threads
       |
       v
Virtual Thread Scheduler
       |
       v
16 Carrier Thread Parallelism
```

You can also configure:

```bash
-Djdk.virtualThreadScheduler.maxPoolSize=256
```

Official Java 21 API documentation: [Java Thread virtual-thread scheduler properties](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/Thread.html?utm_source=chatgpt.com)

# 9. Very important interview point: DB connection pool

Suppose:

```text
100,000 Virtual Threads
```

But HikariCP:

```properties
spring.datasource.hikari.maximum-pool-size=20
```

Then:

```text
100,000 Virtual Threads
          |
          v
Only 20 DB Connections
```

Conceptually:

```text
VT-1  ---> DB Connection-1
VT-2  ---> DB Connection-2
...
VT-20 ---> DB Connection-20

VT-21
VT-22
VT-23
...
      ---> Waiting for DB connection
```

**Virtual Threads do not magically increase database capacity.**

This is a very important 12+ years interview answer:

> **Virtual threads improve thread scalability for blocking I/O workloads, but downstream resources such as database connection pools, HTTP connection pools, Kafka limits, and external service capacity still impose concurrency limits.**

### Your exact question: "How many Virtual Threads per Carrier?"

> **No fixed number. One carrier executes one mounted virtual thread at a time, but over time the same carrier can execute thousands or millions of different virtual threads.**

```text
Carrier-1

Time-1 ---> VT-1
Time-2 ---> VT-50
Time-3 ---> VT-200
Time-4 ---> VT-1000
Time-5 ---> VT-50000
```

**This is the main concept you should remember.**

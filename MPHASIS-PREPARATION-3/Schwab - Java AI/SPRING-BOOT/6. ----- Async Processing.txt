continue without waiting for the method completion

First Enable: @EnableAsync

------------------------------------------------------


The main difference is:

> **Async Execution = how a task runs**
> **Async Processing = how a business request/workflow is handled**

## Async Processing vs Async Execution

| Feature             | Async Execution                                                   | Async Processing                                  |
| ------------------- | ----------------------------------------------------------------- | ------------------------------------------------- |
| Meaning             | Execute a method/task in another thread                           | Process a request independently in the background |
| Focus               | **Thread execution**                                              | **Business workflow / system processing**         |
| Caller waits?       | Usually no                                                        | No                                                |
| Common technology   | `@Async`, `CompletableFuture`, `ExecutorService`, Virtual Threads | Kafka, RabbitMQ, SQS, event-driven architecture   |
| Same application?   | Usually yes                                                       | Can be different microservices                    |
| Durable?            | Usually no                                                        | Usually yes with message broker                   |
| Application crash   | Task may be lost                                                  | Message can remain in broker                      |
| Retry               | Need custom handling                                              | Broker/framework retry can be configured          |
| Microservices usage | Parallel API calls, background methods                            | Order processing, payment workflow, notifications |

# 1. Async Execution

**Definition:**

> Async Execution means executing a method or task on a **different thread without blocking the caller thread**.

### Example

```java
@Service
public class EmailService {

    @Async
    public void sendEmail() {

        System.out.println(
            Thread.currentThread().getName()
        );

        // Send email
    }
}
```

Caller:

```java
@RestController
public class UserController {

    private final EmailService emailService;

    public UserController(EmailService emailService) {
        this.emailService = emailService;
    }

    @PostMapping("/users")
    public String createUser() {

        System.out.println("Create User");

        emailService.sendEmail();

        return "User Created";
    }
}
```

### Execution flow

```text
HTTP Request
     |
     v
Tomcat Thread
http-nio-8080-exec-1
     |
     |---- Create User
     |
     |---- Call sendEmail()
              |
              v
        Async Thread
        taskExecutor-1
              |
              v
         Send Email

Tomcat Thread
     |
     v
Return Response
```

The HTTP thread **does not wait for email completion**.

### Example timeline

```text
0 sec   Create User
1 sec   Start Async Email
1 sec   Return Response

Async Thread
     |
     |---- Email Processing
     |
5 sec   Email Completed
```

### Real-time use cases

* Send email
* Generate report
* Write audit logs
* Parallel API calls
* Background calculations
* File processing

### Technologies

```text
@Async
CompletableFuture
ExecutorService
ThreadPoolExecutor
Virtual Threads
```

# 2. Async Processing

**Definition:**

> Async Processing means accepting a request or event and **processing the business operation independently at a later time or in another service**.

Usually we use a **message broker**.

### Example: Order Processing

```text
Client
   |
   v
Order Service
   |
   |---- Save Order
   |
   |---- Publish ORDER_CREATED
   |
   v
Kafka
   |
   +--------------------+
   |                    |
   v                    v
Payment Service    Inventory Service
   |                    |
Process Payment     Update Stock
```

The Order Service does not directly wait for:

```text
Payment Service
Inventory Service
Notification Service
```

It publishes an event.

```java
kafkaTemplate.send(
    "order-created",
    orderEvent
);
```

Consumer:

```java
@KafkaListener(topics = "order-created")
public void processOrder(OrderEvent event) {

    processPayment(event);
}
```

### Execution flow

```text
Request
   |
   v
Order Service
   |
   | Save Order
   |
   | Publish Kafka Event
   |
   v
Response
"Order Accepted"


Later / Independently

Kafka
   |
   v
Payment Consumer
   |
   v
Process Payment
```

This is **Async Processing**.

# Interview Answer

You can explain it like this:

> **Async Execution is a concurrency technique where a task is executed on a separate thread so that the caller thread is not blocked. Examples are `@Async`, `CompletableFuture`, and `ExecutorService`.**
>
> **Async Processing is an architectural approach where business operations are processed independently using messaging or event-driven systems like Kafka or RabbitMQ. It provides loose coupling, scalability, durability, and retry capabilities.**

## One-line difference

```text
Async Execution  = Different Thread
Async Processing = Independent Business Processing
```

### Easy way to remember

```text
@Async              → Async Execution
CompletableFuture   → Async Execution

Kafka               → Async Processing
RabbitMQ            → Async Processing
SQS                 → Async Processing
```

**Important interview point:** Kafka consumer processing may internally also use **async execution/thread pools**. So **Async Processing can use Async Execution internally**, but both concepts are not the same.

2. @Async with Custom Thread Pool

By default Spring uses SimpleAsyncTaskExecutor, which creates new threads frequently. In production, use a thread pool.

Configuration
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean(name = "customExecutor")
    public Executor customExecutor() {

        ThreadPoolTaskExecutor executor =
                new ThreadPoolTaskExecutor();

        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("ASYNC-");
        executor.initialize();

        return executor;
    }
}

Usage
@Service
public class ReportService {

    @Async("customExecutor")
    public void generateReport() {

        System.out.println(
            "Running on: " + Thread.currentThread().getName());

        // Business logic
    }
}

Output
Running on: ASYNC-1

Thread Pool Parameters
Parameter	DescriptionCorePoolSize	Minimum threads always alive
MaxPoolSize	Maximum thread limit
QueueCapacity	Tasks waiting in queue
ThreadNamePrefix	Custom thread names
Benefits
Better resource management
Controls thread creation
Suitable for production environments
3. @Async with CompletableFuture

When the caller needs the result of an async operation, use CompletableFuture.

Service
@Service
public class UserService {

    @Async
    public CompletableFuture<String> getUserDetails() {

        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }

        return CompletableFuture.completedFuture(
                "User Details Loaded");
    }
}

Controller
@RestController
public class UserController {

    @Autowired
    private UserService userService;

    @GetMapping("/user")
    public String user() throws Exception {

        CompletableFuture<String> future =
                userService.getUserDetails();

        System.out.println("Doing other work...");

        return future.get();
    }
}

Multiple Async Calls with CompletableFuture
@Async
public CompletableFuture<String> getUser() {
    return CompletableFuture.completedFuture("User");
}

@Async
public CompletableFuture<String> getOrders() {
    return CompletableFuture.completedFuture("Orders");
}

Combine Results
CompletableFuture<String> user =
        service.getUser();

CompletableFuture<String> orders =
        service.getOrders();

CompletableFuture.allOf(user, orders).join();

String result =
        user.get() + " - " + orders.get();

Output
User - Orders

Benefits
Return results asynchronously
Combine multiple tasks
Better than plain Future
4. @Async with RestTemplate

Useful when making external API calls asynchronously.

Configuration
@Bean
public RestTemplate restTemplate() {
    return new RestTemplate();
}

@Service
public class ApiService {

    @Autowired
    private RestTemplate restTemplate;

    @Async
    public CompletableFuture<String> callExternalApi() {

        String response =
                restTemplate.getForObject(
                        "https://jsonplaceholder.typicode.com/posts/1",
                        String.class);

        return CompletableFuture.completedFuture(response);
    }
}

Controller
@RestController
public class ApiController {

    @Autowired
    private ApiService apiService;

    @GetMapping("/api")
    public String test() throws Exception {

        CompletableFuture<String> response =
                apiService.callExternalApi();

        return response.get();
    }
}

Multiple REST Calls in Parallel
CompletableFuture<String> api1 =
        apiService.callApi1();

CompletableFuture<String> api2 =
        apiService.callApi2();

CompletableFuture<String> api3 =
        apiService.callApi3();

CompletableFuture.allOf(api1, api2, api3).join();

String result =
        api1.get() +
        api2.get() +
        api3.get();


This significantly reduces total response time because APIs execute concurrently.

# 1. Synchronous Execution

In synchronous processing, the **same thread waits until the task completes**.

### Example

```text
Client
  |
  | Request
  v
Tomcat Thread-1
  |
  |---- sendEmail() -------- 5 sec
  |
  |---- generateReport() --- 10 sec
  |
  v
Response

Total time = 15 seconds
```

### Code

```java
@RestController
public class OrderController {

    @GetMapping("/order")
    public String createOrder() {

        sendEmail();        // 5 seconds
        generateReport();   // 10 seconds

        return "Order Created";
    }
}
```

### Execution

```text
Thread: http-nio-8080-exec-1

sendEmail()
     ↓
wait 5 sec

generateReport()
     ↓
wait 10 sec

return response
```

The request thread is **blocked for 15 seconds**.

---

# 2. Async Execution

In asynchronous execution, the request thread **delegates the task to another thread**.

```text
Client
  |
  v
Tomcat Thread-1
  |
  |---- createOrder()
  |
  |---- submit async task
  |             |
  |             v
  |        Async Thread-1
  |             |
  |             |---- sendEmail()
  |             |
  |             |---- generateReport()
  |
  v
Response immediately
```

Example:

```text
Tomcat Thread
http-nio-8080-exec-1
        |
        | submit task
        v
Async Thread
taskExecutor-1
```

The Tomcat thread **does not wait**.

# 3. Spring Boot `@Async`

Spring provides the `@Async` annotation for asynchronous method execution.

## Step 1: Enable Async

```java
@SpringBootApplication
@EnableAsync
public class OrderApplication {

    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

`@EnableAsync` tells Spring:

> Find methods annotated with `@Async` and execute them asynchronously.

---

## Step 2: Create Async Service

```java
@Service
public class EmailService {

    @Async
    public void sendEmail() {

        System.out.println(
            "Thread: " + Thread.currentThread().getName()
        );

        try {
            Thread.sleep(5000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        System.out.println("Email Sent");
    }
}
```

---

## Step 3: Call Async Method

```java
@RestController
public class OrderController {

    private final EmailService emailService;

    public OrderController(EmailService emailService) {
        this.emailService = emailService;
    }

    @PostMapping("/order")
    public String createOrder() {

        System.out.println(
            "Controller Thread: "
            + Thread.currentThread().getName()
        );

        emailService.sendEmail();

        return "Order Created";
    }
}
```

# 7. How ThreadPoolTaskExecutor Handles 10 Requests

Suppose:

```text
Core Pool Size = 5
Max Pool Size  = 10
Queue Capacity = 100
```

10 async requests arrive.

### Requests 1–5

```text
Request-1 → Thread-1
Request-2 → Thread-2
Request-3 → Thread-3
Request-4 → Thread-4
Request-5 → Thread-5
```

All core threads are busy.

### Requests 6–10

Because queue capacity is `100`:

```text
Request-6  → Queue
Request-7  → Queue
Request-8  → Queue
Request-9  → Queue
Request-10 → Queue
```

Important interview point:

> **Spring does not immediately create threads 6–10.**

Why?

Because the queue still has capacity.

```text
Core Threads Full
       |
       v
Queue Task
       |
       v
Queue Full?
   |
   | NO
   v
Wait in Queue
```

Only when the **queue becomes full** will Spring create threads beyond the core pool size, up to `maxPoolSize`.

# 10. Run Multiple APIs in Parallel

This is a **real-time microservice scenario**.

Suppose Order Service calls:

```text
Customer Service → 5 sec
Payment Service  → 5 sec
Inventory Service → 5 sec
```

Synchronous:

```text
Customer   5 sec
    ↓
Payment    5 sec
    ↓
Inventory  5 sec

Total = 15 seconds
```

Async parallel execution:

```text
             Customer API
             Thread-1
             5 sec
            /
Request --- Payment API
            Thread-2
            5 sec
            \
             Inventory API
             Thread-3
             5 sec
```

Total approximately:

```text
5 seconds
```

Code:

```java
CompletableFuture<Customer> customerFuture =
        CompletableFuture.supplyAsync(
                () -> customerService.getCustomer()
        );

CompletableFuture<Payment> paymentFuture =
        CompletableFuture.supplyAsync(
                () -> paymentService.getPayment()
        );

CompletableFuture<Inventory> inventoryFuture =
        CompletableFuture.supplyAsync(
                () -> inventoryService.getInventory()
        );

CompletableFuture.allOf(
        customerFuture,
        paymentFuture,
        inventoryFuture
).join();
```

# Async Processing vs Async Execution

| Async Processing                                | Async Execution                 |
| ----------------------------------------------- | ------------------------------- |
| Architecture concept                            | Code execution technique        |
| Background processing                           | Executes code on another thread |
| Kafka                                           | `@Async`                        |
| RabbitMQ                                        | `CompletableFuture`             |
| Event-driven architecture                       | `ExecutorService`               |
| Tasks may execute later                         | Usually starts immediately      |
| Can survive service restart with durable broker | In-memory task can be lost      |
| Microservice-level                              | JVM/thread-level                |

### Example Async Processing

```text
Order Service
     |
     | OrderCreated Event
     v
Kafka
     |
     +------> Email Service
     |
     +------> Inventory Service
     |
     +------> Notification Service
```

### Example Async Execution

```text
Order Service JVM
       |
       v
@Async
       |
       v
ThreadPoolTaskExecutor
       |
       +-- Thread-1
       +-- Thread-2
       +-- Thread-3
```

# Interview Answer

> **Asynchronous execution means executing a task on a separate thread so that the calling thread is not blocked. In Spring Boot, we can implement it using `@EnableAsync` and `@Async`. Spring uses a TaskExecutor to submit async tasks to a thread pool. For production applications, we configure ThreadPoolTaskExecutor with core pool size, maximum pool size, and queue capacity.**
>
> **Async processing is a broader architectural concept. In microservices, Kafka or RabbitMQ can be used for durable asynchronous processing, whereas `@Async` provides asynchronous execution within the same JVM.**

**For your 10+ years Spring Boot/Microservices interview preparation, the next important topic is: `@Async` internal proxy working + self-invocation problem + exception handling + `@Transactional` with `@Async`.**

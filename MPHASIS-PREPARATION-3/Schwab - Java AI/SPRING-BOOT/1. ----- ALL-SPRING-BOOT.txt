## 4. Difference between `@Profile` and `@Primary`

| @Profile                       | @Primary                                 |
| ------------------------------ | ---------------------------------------- |
| Controls bean creation         | Controls bean selection                  |
| Environment based              | Dependency Injection based               |
| Bean may not exist             | Bean exists                              |
| Used for dev/test/prod configs | Used when multiple implementations exist |


Q4. When should you use profiles?

	Best Use Cases:

		Different database configurations

		External service endpoints

		Logging levels

		Security settings

		Cache providers

`@Profile` is **not used on fields, constructors, or setters**.

**Question:** Where can `@Profile` be applied?

**Answer:**
"`@Profile` can be applied at the class level (such as `@Component`, `@Service`, `@Repository`, `@Configuration`) 
and at the `@Bean` method level. It cannot be applied to fields, constructors, or setter methods because 
it controls bean 
registration in the Spring container, not dependency injection points."

-----------------------------------------------------------Circuit Breaker--------------------------------------

1. What is it?

	Circuit Breaker is a design pattern used in microservices to prevent cascading failures when one service is 
	unavailable, slow, or throwing errors.
	
2. Why do we need it?

	Problems:

		Increased response time
		Thread exhaustion
		Resource wastage
		Entire application becomes slow

	Circuit Breaker prevents this.

	Problems:

		Threads are blocked.
		Connection pool becomes full.
		CPU usage increases.
		Memory usage increases.
		Order Service may also crash.
		Cascading failure occurs.

	This is where Circuit Breaker is useful.
	
3. How it works?

	Circuit Breaker has 3 states. OPEN -> HALF-OPEN->CLOSED

5. Benefits

		Prevents cascading failures.
		Improves system stability.
		Faster response during failures.
		Reduces unnecessary network calls.
		Better user experience.
		Supports automatic recovery.
		
7. Monitoring

		Resilience4j integrates with:

		Spring Boot Actuator
		Prometheus
		Grafana

		Metrics:

			Successful Calls
			Failed Calls
			Slow Calls
			Open State Duration
			Circuit Status

		Endpoint:

		/actuator/circuitbreakers

resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        minimumNumberOfCalls: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3		
		
| Property                              | Meaning                  |
| ------------------------------------- | ------------------------ |
| slidingWindowSize                     | Monitor last 10 requests |
| minimumNumberOfCalls                  | Need at least 10 calls   |
| failureRateThreshold                  | Open if failure >= 50%   |
| waitDurationInOpenState               | Stay OPEN for 10 seconds |
| permittedNumberOfCallsInHalfOpenState | Allow 3 test requests    |

---------------------------------------------------------Bulkhead Patteren----------------------------------


What is Bulkhead Pattern?

	Bulkhead Pattern is a resilience pattern used in microservices to isolate resources between different services or operations 
	so that failure or overload in one component does not affect others.
	
	Circuit Breaker: Service Failure
	
	Bulkhead Pattern: Resource Exhaustion
	
Two types of Bulkhead Patteren.

1. Semaphore Bulkhead

		 @Bulkhead(
					name = "paymentBulkhead",
					type = Bulkhead.Type.SEMAPHORE,
					fallbackMethod = "bulkheadFallback"
			)
			public String processPayment(Long id)
					throws InterruptedException {

				Thread.sleep(10000);

				return "Payment Success : " + id;
			}
			
Configuration
resilience4j:
  bulkhead:
    instances:
      paymentBulkhead:
        maxConcurrentCalls: 2
        maxWaitDuration: 1s	

2. Thread Pool Bulkhead

	Thread Pool Bulkhead Example
	@Bulkhead(
		 name="paymentBulkhead",
		 type = Bulkhead.Type.THREADPOOL,
		 fallbackMethod="bulkheadFallback")

	Configuration:

		resilience4j:
		  thread-pool-bulkhead:
			instances:
			  paymentBulkhead:
				coreThreadPoolSize: 2
				maxThreadPoolSize: 4
				queueCapacity: 10

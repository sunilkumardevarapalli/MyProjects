# 1. What is Circuit Breaker?

A **Circuit Breaker is a fault-tolerance design pattern used to prevent repeated calls to a failing downstream service**.

### Simple definition for interview

> Circuit Breaker monitors calls to a downstream service. When failures cross a configured threshold, it stops calling the failed service temporarily and moves to OPEN state. After a waiting period, it allows limited test calls in HALF_OPEN state. If the service recovers, it moves back to CLOSED.

Suppose:

```text
Order Service
      |
      | REST API
      v
Payment Service
```

Order Service calls Payment Service:

```text

```

Now Payment Service has a problem:

```text

Network Issue
```

Without Circuit Breaker:

```text

Request 2  ---> Payment Service ---> FAIL
Request 3  ---> Payment Service ---> FAIL
Request 4  ---> Payment Service ---> FAIL
Request 5  ---> Payment Service ---> FAIL

Request 10000 ---> Payment Service ---> FAIL
```

Every request is hitting the already failed service.

Problems:

* Threads are blocked.
* Connection pool becomes full.
* CPU usage increases.
* Memory usage increases.
* Order Service may also crash.
* Cascading failure occurs.

This is where **Circuit Breaker** is useful.

# 3. Circuit Breaker has 3 main states

```text
             Failure threshold reached
     +-----------------------------------+
     |                                   |
     v                                   |
+----------+                         +--------+
|  CLOSED  | ----------------------> |  OPEN  |
+----------+                         +--------+
     ^                                   |
     |                                   |
     | Success                           | Wait Duration
     |                                   |
     |                                   v
     |                            +-------------+
     +----------------------------| HALF_OPEN   |
                                  +-------------+
                                         |
                                         | Failure
                                         v
                                      OPEN
```

The states are:

```text
1. CLOSED
2. OPEN
3. HALF_OPEN
```

```xml
    <groupId>io.github.resilience4j</groupId>
```

Service:

```java
@Service
public class PaymentClient {

    private final RestTemplate restTemplate;

    public PaymentClient(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    @CircuitBreaker(
        name = "paymentService",
        fallbackMethod = "paymentFallback"
    )
    public String processPayment() {

        return restTemplate.getForObject(
            "http://payment-service/payment",
            String.class
        );
    }

    public String paymentFallback(Throwable ex) {

        return "Payment Service temporarily unavailable";
    }
}
```

# Very important interview question

### If 10 requests come simultaneously, will Circuit Breaker process one by one?

**No.**

Requests can execute concurrently:

```text
Thread-1 ---> Request 1
Thread-2 ---> Request 2
Thread-3 ---> Request 3
Thread-4 ---> Request 4
...
Thread-10 --> Request 10
```

Resilience4j records results in the sliding window.

```text
Request completed ---> Record SUCCESS / FAILURE
```

When enough calls are recorded:

```text
minimumNumberOfCalls = 10
```

it calculates:

```text
Failure Rate
```

If:

```text
Failure Rate >= Threshold
```

then:

```text
CLOSED ---> OPEN
```

**Important real-time point:** some requests already permitted while the breaker was CLOSED may still be executing when another request completion causes the state to change to OPEN. OPEN blocks **new permission requests**; it does not normally cancel already-running HTTP calls.

# Interview answer in 30 seconds

> Circuit Breaker has CLOSED, OPEN and HALF_OPEN states. In CLOSED state, all calls are allowed and the Circuit Breaker records success and failure. For example, if the sliding window size and minimum calls are 10 and 6 out of 10 calls fail, the failure rate is 60%. If the configured threshold is 50%, the Circuit Breaker moves to OPEN. In OPEN state, downstream calls are rejected and fallback is executed. After the configured wait duration, it moves to HALF_OPEN and permits limited test calls, for example 3 calls. If their failure rate is below the threshold, it moves to CLOSED. Otherwise, it moves back to OPEN.

**One small correction that is very important for senior-level interviews:** `slidingWindowSize = 10` does **not mean "wait for exactly 10 new requests forever."** In `COUNT_BASED`, it maintains the **latest 10 recorded calls as a sliding window**. I can explain **sliding window with R1–R20 step by step**, including how old requests are removed—this is a common 10–12 years experience interview question.

4. Real-Time Project Use Case

In my project, Order Service was dependent on Payment Service and Notification Service. 
If Payment Service became unavailable, user requests started taking a long time due to repeated retries. 
To avoid cascading failures, we implemented Resilience4j Circuit Breaker. When a predefined failure threshold was reached,
 the circuit moved to Open state and immediately returned a fallback response instead of calling the unavailable service. 
 Once Payment Service recovered, the circuit automatically transitioned through Half-Open to Closed state.

Interview Answer (2 Minutes)

Circuit Breaker is a fault-tolerance pattern used in microservices to prevent cascading failures when dependent services become slow or unavailable. In our project, Order Service depended on Payment Service. To avoid continuous failed calls and resource exhaustion, we implemented Resilience4j Circuit Breaker. Under normal conditions, the circuit remains in Closed state and requests flow normally. If the failure rate exceeds a configured threshold, the circuit moves to Open state and immediately returns a fallback response without calling the downstream service. After a configured wait time, it enters Half-Open state and allows a few test requests. If those requests succeed, the circuit closes; otherwise, it reopens. We monitored circuit breaker metrics using Spring Boot Actuator, Prometheus, and Grafana. This improved system resilience, reduced response times during outages, and prevented cascading failures across microservices.

# `@Profile` in Spring Boot – Complete Interview Notes (12+ Years Experience)

## 1. Purpose of `@Profile`

`@Profile` is used for **conditional bean registration** based on the active environment.

Common environments:

* dev
* test
* stage
* prod
* cloud

Example:

```java
@Service
@Profile("dev")
public class MockPaymentService implements PaymentService {
}
```

```java
@Service
@Profile("prod")
public class RealPaymentService implements PaymentService {
}
```

Only the bean matching the active profile is created.

---

# 2. Where Can `@Profile` Be Declared?

### ✅ Class Level

```java
@Service
@Profile("dev")
public class DevEmailService {
}
```

Applicable on:

* `@Component`
* `@Service`
* `@Repository`
* `@Controller`
* `@RestController`

---

### ✅ Configuration Class Level

```java
@Configuration
@Profile("prod")
public class ProdConfig {
}
```

All beans inside this configuration load only for `prod`.

---

### ✅ `@Bean` Method Level

```java
@Bean
@Profile("test")
public DataSource dataSource() {
    return new EmbeddedDatabaseBuilder().build();
}
```

---

### ❌ Not Supported

#### Field Level

```java
@Autowired
@Profile("dev")   // Invalid
private EmailService emailService;
```

#### Constructor Level

```java
@Profile("dev")   // Invalid
public EmailService() {
}
```

#### Setter Level

```java
@Profile("dev")   // Invalid
public void setEmailService(EmailService emailService) {
}
```

**Reason:** `@Profile` controls bean registration, not dependency injection.

# 5. Bean Registration Priority / Invocation Order

This is a very common senior-level interview question.

## Spring Bean Resolution Order

### Highest → Lowest

### 1. `@Profile`

Checks whether bean should exist.

```java
@Service
@Profile("dev")
public class DevService {
}
```

If profile doesn't match:

```text
Bean is NOT created.
```

---

### 2. `@Conditional`

Checks custom conditions.

```java
@ConditionalOnProperty(
   name="sms.enabled",
   havingValue="true"
)
```

If condition fails:

```text
Bean is NOT created.
```

---

### 3. `@Primary`

Chooses default bean among existing beans.

```java
@Service
@Primary
public class EmailServiceImpl {
}
```

---

### 4. `@Qualifier`

Explicitly selects bean.

```java
@Autowired
@Qualifier("smsService")
private NotificationService service;
```

---

## Final Priority Order

```text
1. @Profile
2. @Conditional
3. @Primary
4. @Qualifier
```

### Important

If `@Profile` fails:

```text
Bean never enters Spring Container.
```

Then:

```text
@Primary → Not Checked
@Qualifier → Not Checked
```

because bean doesn't exist.

# Senior-Level Interview Answer (1 Minute)

> "`@Profile` is a conditional bean registration mechanism in Spring used to load environment-specific beans such as dev, 
test, stage, and prod configurations. 

It can be applied at class, configuration, and `@Bean` method levels. 

Multiple profiles
 and profile expressions (AND, OR, NOT) are supported. 
 
 During startup, Spring first evaluates `@Profile`; 
 if the profile doesn't match, the bean is never created. 
 
 Therefore, `@Profile` has higher precedence than 
 `@Primary` and `@Qualifier`. 
 
 Profiles can be activated through application properties, command-line arguments, 
 JVM properties, environment variables, IDE settings, programmatically, and `@ActiveProfiles` for testing.
 Profile-specific files like `application-dev.properties` override values from `application.properties`."

In **Spring Framework**, **Profile-Based Annotations** are used to enable or disable beans and configurations based on 
the active environment (profile), such as `dev`, `test`, or `prod`.

The primary annotation is **`@Profile`**.

### Example: Class-Level Usage


import org.springframework.context.annotation.Profile;
import org.springframework.stereotype.Service;

@Service
@Profile("dev")
public class DevEmailService implements EmailService {
    // implementation
}


This bean is created only when the `dev` profile is active.

---

### Configuration Class Example


@Configuration
@Profile("prod")
public class ProdConfig {

    @Bean
    public DataSource dataSource() {
        // production datasource
    }
}


The entire configuration is loaded only for the `prod` profile.

---

### Multiple Profiles


@Service
@Profile({"dev", "test"})
public class MockEmailService implements EmailService {
}


The bean is loaded if either `dev` or `test` is active.

---

### Profile Expressions


@Service
@Profile("!prod")
public class LocalEmailService implements EmailService {
}


Loaded when the active profile is **not** `prod`.

You can also use expressions:


@Profile("dev & cloud")



@Profile("dev | test")

**application.properties**

* ✅ Class level (`@Component`, `@Service`, `@Repository`, etc.)
* ✅ `@Configuration` class level
* ✅ `@Bean` method level

Example:

while `@Primary` is used to resolve ambiguity when **multiple beans of the same type are active**.

## 3. What is the priority of `@Profile`?

**Answer:**

`@Profile` is evaluated during bean registration.

If a profile does not match:

* Bean is never registered.
* Bean does not exist in Spring Context.
* `@Primary` and `@Qualifier` are not considered because the bean isn't created.

Example:

```java
@Service
@Profile("dev")
@Primary
public class DevService {
}
```

If `dev` profile is not active:

```java
DevService
```

is not registered at all.

**Priority Order:**

1. `@Profile` decides whether bean exists.
2. `@Conditional` decides whether bean exists.
3. `@Primary` resolves conflicts among existing beans.
4. `@Qualifier` provides explicit bean selection.

# Senior-Level Interview Answer

> Spring Boot first loads `application.properties`. If it finds `spring.profiles.active=dev`, 
it activates the `dev` profile and then loads `application-dev.properties`. 
Profiles can be activated through `application.properties`, command-line arguments, JVM system properties,
 environment variables, programmatic configuration, IDE settings, and `@ActiveProfiles` for testing. Among these, 
 command-line arguments have the highest precedence.

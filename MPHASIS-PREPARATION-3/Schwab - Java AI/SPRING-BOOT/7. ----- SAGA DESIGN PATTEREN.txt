

SAGA pattern (event-driven)** for an **Order system**

	Which is used to manage and maintain the data consistency across multiple MS's.

	Each transactions executed by a single MS and status changes and send to other services 

	via serviceMesh/ServiceComunicator/Broker.SAGA involved in this statge.

	we can acchive SAGA Design Patteren in two ways 

	1) Choreography-based Saga (event-driven)
	
		Task executed independently once the task is completed it invokes the next task in the sequence
		till the usecase completed. We can N number of tasks in this use case.
		
		if any task fail then invoke compoenstating for previous communication.
	
	2) Orchastation
	
We’ll implement:

	✅ Choreography-based Saga (event-driven)
	✅ Services: Order, Payment, Inventory
	✅ Kafka (or any broker) for events
	✅ Compensation logic (rollback)
 
# 🔥 1. Problem Statement (Real Scenario)

	When a user places an order:

		1. Create Order → `PENDING`
		2. Reserve Inventory
		3. Process Payment
		4. Confirm Order

If any step fails → rollback previous steps
 
# 🧠 2. Saga Flow (Event-Based)

Order Created → Inventory Reserved → Payment Completed → Order Confirmed
         ↓              ↓                  ↓
      (fail)         (fail)            (fail)
         ↓              ↓                  ↓
   Cancel Order ← Release Stock ← Refund Payment

## Business Logic

```java
@Service
public class OrderService {

    @Autowired
    private KafkaTemplate<String, Object> kafka;

    @Autowired
    private OrderRepository repo;

    public String createOrder() {
        Order order = new Order();
        order.setId(UUID.randomUUID().toString());
        order.setStatus("PENDING");
        repo.save(order);

        kafka.send("order-created", new OrderCreatedEvent(order.getId()));

        return order.getId();
    }
}
```

---

## Event Listeners (Saga Orchestration Logic)

```java
@Component
public class OrderEventListener {

    @Autowired
    private OrderRepository repo;

    @Autowired
    private KafkaTemplate<String, Object> kafka;

    @KafkaListener(topics = "payment-success")
    public void handlePaymentSuccess(PaymentSuccessEvent event) {
        Order order = repo.findById(event.getOrderId()).get();
        order.setStatus("COMPLETED");
        repo.save(order);

        kafka.send("order-completed", new OrderCompletedEvent(order.getId()));
    }

    @KafkaListener(topics = "payment-failed")
    public void handlePaymentFailure(PaymentFailedEvent event) {
        Order order = repo.findById(event.getOrderId()).get();
        order.setStatus("CANCELLED");
        repo.save(order);

        kafka.send("order-cancelled", new OrderCancelledEvent(order.getId()));
    }
}
```

        return new DefaultKafkaProducerFactory<>(Map.of(

# 🧠 10. Key Concepts (Senior-Level Insights)

### ✅ Why Saga instead of @Transactional?

* Distributed system → multiple DBs
* No global transaction support
* Avoid 2PC (Two Phase Commit)

---

### ✅ Choreography vs Orchestration

| Type          | Description                             |
| ------------- | --------------------------------------- |
| Choreography  | Services react to events (we used this) |
| Orchestration | Central coordinator controls flow       |

---

### ✅ Real Production Improvements

1. Add **Retry + DLQ (Dead Letter Queue)**
2. Use **Idempotency keys**
3. Add **Event versioning**
4. Store **Saga state (DB or Redis)**
5. Use **Outbox pattern** (avoid message loss)

# 🧠 1. CHOREOGRAPHY (Event-Driven)

👉 Think: **“No boss — everyone reacts to events”**

Each service listens and reacts.

---

## 📦 Example: Order Flow (Choreography)

### Flow

```text
Order Created → Inventory Service reacts
Inventory Reserved → Payment Service reacts
Payment Success → Order Service reacts
```

## 🚨 Problem in Choreography

👉 Who controls the flow? **No one**

👉 If something fails:

* Multiple services must handle compensation
* Hard to track flow
* Debugging nightmare

# 🧠 2. ORCHESTRATION (Central Controller)

👉 Think: **“There is a boss controlling everything”**

One service = **Orchestrator**

---

## 📦 Example: Order Flow (Orchestration)

### Flow

```text
Orchestrator → Call Inventory
Orchestrator → Call Payment
Orchestrator → Confirm Order
```

---

## 🧩 Orchestrator Logic

```java
@Service
public class OrderSagaOrchestrator {

    @Autowired
    private KafkaTemplate<String, Object> kafka;

    public void startSaga(String orderId) {
        kafka.send("reserve-inventory", orderId);
    }

    @KafkaListener(topics = "inventory-success")
    public void onInventorySuccess(String orderId) {
        kafka.send("process-payment", orderId);
    }

    @KafkaListener(topics = "payment-success")
    public void onPaymentSuccess(String orderId) {
        kafka.send("complete-order", orderId);
    }

    @KafkaListener(topics = "payment-failed")
    public void onPaymentFailure(String orderId) {
        kafka.send("release-inventory", orderId);
        kafka.send("cancel-order", orderId);
    }
}
```

# ⚖️ Real Difference (Important Insight)

## 🔹 Choreography

✔ Easy to start
✔ Fully decoupled
❌ Hidden flow
❌ Hard debugging
❌ Event chain becomes spaghetti

👉 Best for:

* Simple workflows
* Few services

---

## 🔹 Orchestration

✔ Clear control
✔ Easy debugging
✔ Complex logic manageable
❌ Central dependency (orchestrator)

👉 Best for:

* Complex business flows
* Many services
* Real enterprise systems

---

# 🔥 Real Production Advice (12+ yrs level)

👉 Most companies start with **Choreography**
👉 Then migrate to **Orchestration** when system grows

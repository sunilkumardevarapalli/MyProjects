What is @RequiredArgsConstructor?

@RequiredArgsConstructor is a Lombok annotation that generates a constructor for all final and @NonNull fields. 
It is commonly used in Spring Boot for constructor-based dependency injection, reducing boilerplate code and making dependencies 
immutable.

Conditional Beans in Spring Boot

Conditional Beans allow Spring Boot to create or skip bean registration based on certain conditions. This is very useful for:

Environment-specific configurations
Feature toggles
Auto-configuration
Optional dependencies
Different implementations for different profiles

Spring Boot provides several @Conditional... annotations.

1. @ConditionalOnProperty

Creates a bean only if a property exists and matches a value.

@Configuration
public class AppConfig {

    @Bean
    @ConditionalOnProperty(
        name = "feature.email.enabled",
        havingValue = "true"
    )
    public EmailService emailService() {
        return new EmailService();
    }
}


application.properties

feature.email.enabled=true


Bean is created only when the property value is true.

2. @ConditionalOnMissingBean

Creates a bean only when no bean of the same type already exists.

@Bean
@ConditionalOnMissingBean
public NotificationService notificationService() {
    return new DefaultNotificationService();
}


Commonly used in Spring Boot auto-configurations to provide default implementations.

3. @ConditionalOnBean

Creates a bean only if another bean is present.

@Bean
@ConditionalOnBean(DataSource.class)
public DatabaseHealthCheck databaseHealthCheck() {
    return new DatabaseHealthCheck();
}


This bean will be registered only if a DataSource bean exists.

4. @ConditionalOnClass

Creates a bean only when a class is available in the classpath.

@Bean
@ConditionalOnClass(name = "com.mysql.cj.jdbc.Driver")
public MySqlConfiguration mySqlConfiguration() {
    return new MySqlConfiguration();
}


Useful for optional libraries.

5. @ConditionalOnMissingClass

Creates a bean only when a class is NOT present.

@Bean
@ConditionalOnMissingClass("com.example.RedisClient")
public InMemoryCache cache() {
    return new InMemoryCache();
}

6. @ConditionalOnExpression

Evaluates a SpEL expression.

@Bean
@ConditionalOnExpression("'${app.mode}'.equals('PROD')")
public ProdService prodService() {
    return new ProdService();
}

7. @ConditionalOnResource

Creates a bean only if a resource exists.

@Bean
@ConditionalOnResource(
    resources = "classpath:custom-config.properties")
public CustomLoader customLoader() {
    return new CustomLoader();
}

8. @ConditionalOnWebApplication

Loads bean only in web applications.

@Bean
@ConditionalOnWebApplication
public WebConfig webConfig() {
    return new WebConfig();
}

9. @ConditionalOnNotWebApplication

Loads bean only in non-web applications.

@Bean
@ConditionalOnNotWebApplication
public BatchProcessor batchProcessor() {
    return new BatchProcessor();
}

10. Custom Condition (@Conditional)

You can create your own condition.

Step 1: Create Condition
public class CustomCondition implements Condition {

    @Override
    public boolean matches(
            ConditionContext context,
            AnnotatedTypeMetadata metadata) {

        String env =
                context.getEnvironment().getProperty("app.env");

        return "prod".equalsIgnoreCase(env);
    }
}

Step 2: Use It
@Bean
@Conditional(CustomCondition.class)
public ProdOnlyService prodOnlyService() {
    return new ProdOnlyService();
}

Real-world Example

Suppose you have two payment providers:

public interface PaymentService {
    void pay();
}

Razorpay Implementation
@Service
@ConditionalOnProperty(
    name = "payment.provider",
    havingValue = "razorpay")
public class RazorpayService implements PaymentService {

    public void pay() {
        System.out.println("Razorpay Payment");
    }
}

Stripe Implementation
@Service
@ConditionalOnProperty(
    name = "payment.provider",
    havingValue = "stripe")
public class StripeService implements PaymentService {

    public void pay() {
        System.out.println("Stripe Payment");
    }
}


application.properties

payment.provider=razorpay


Only RazorpayService will be registered as a Spring bean.

Interview Questions
Difference between @Profile and @ConditionalOnProperty?

@Profile

@Profile("prod")

Environment-based bean loading.
Used for Dev, QA, Prod separation.

@ConditionalOnProperty

@ConditionalOnProperty(name="feature.enabled")

Property-based bean loading.
Used for feature flags and configuration switches.
Why is @ConditionalOnMissingBean widely used?

It allows Spring Boot auto-configuration classes to provide default beans that users can override by defining their own beans.

Example:

@Bean
@ConditionalOnMissingBean
public ObjectMapper objectMapper() {
    return new ObjectMapper();
}


If the application defines its own ObjectMapper, Spring Boot skips this bean.

Summary
Annotation	Purpose@ConditionalOnProperty	Based on property value
@ConditionalOnBean	Bean exists
@ConditionalOnMissingBean	Bean does not exist
@ConditionalOnClass	Class present in classpath
@ConditionalOnMissingClass	Class absent in classpath
@ConditionalOnExpression	SpEL expression
@ConditionalOnResource	Resource exists
@ConditionalOnWebApplication	Web application
@ConditionalOnNotWebApplication	Non-web application
@Conditional	Custom condition

Most commonly used in real Spring Boot projects:
 @ConditionalOnProperty, @ConditionalOnMissingBean, @ConditionalOnBean, and @Profile.

If you specifically want Environment-Specific Configuration using @Conditional (custom condition) instead of @Profile, here's a practical example.

Scenario

Load a bean only when the application is running in the production environment.

Step 1: Create a Custom Condition
import org.springframework.context.annotation.Condition;
import org.springframework.context.annotation.ConditionContext;
import org.springframework.core.type.AnnotatedTypeMetadata;

public class ProdEnvironmentCondition implements Condition {

    @Override
    public boolean matches(
            ConditionContext context,
            AnnotatedTypeMetadata metadata) {

        String env = context.getEnvironment()
                            .getProperty("app.environment");

        return "prod".equalsIgnoreCase(env);
    }
}

Step 2: Use @Conditional
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Conditional;
import org.springframework.context.annotation.Configuration;

@Configuration
public class StorageConfig {

    @Bean
    @Conditional(ProdEnvironmentCondition.class)
    public StorageService storageService() {
        return new AwsS3StorageService();
    }
}

Step 3: Set Property

application.properties

app.environment=prod


✅ Bean gets created.

If:

app.environment=dev


❌ Bean is not created.

Multiple Environment Example
Dev Condition
public class DevCondition implements Condition {

    @Override
    public boolean matches(
            ConditionContext context,
            AnnotatedTypeMetadata metadata) {

        return "dev".equalsIgnoreCase(
            context.getEnvironment()
                   .getProperty("app.environment"));
    }
}

Dev Bean
@Bean
@Conditional(DevCondition.class)
public StorageService localStorageService() {
    return new LocalStorageService();
}

Prod Bean
@Bean
@Conditional(ProdEnvironmentCondition.class)
public StorageService awsStorageService() {
    return new AwsS3StorageService();
}

Why use @Conditional when @Profile exists?

@Profile works only with Spring profiles:

@Profile("prod")


@Conditional is more flexible because you can check:

Properties
Environment variables
OS type
Java version
Presence of files
Custom business rules

Example:

return env.equals("prod") && region.equals("us-east");


This kind of logic is not possible directly with @Profile.

Interview Answer

For environment-specific configurations, @Profile is the preferred and simplest approach. @Conditional is used when the decision requires custom logic beyond profile matching, such as checking multiple properties, regions, OS, or external conditions.

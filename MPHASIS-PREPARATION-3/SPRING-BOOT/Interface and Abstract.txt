Interface Definition

	An interface is a blueprint that defines what a class should do. 
	It contains method declarations (contracts), and implementing classes provide the implementation.

	Java 8 onwards:

		Default methods
		Static methods

	Java 9 onwards:

		Private methods
		
		interface PaymentService {
			void pay();
			void refund();
		}
		class UpiPayment implements PaymentService

		class CardPayment implements PaymentService

		class NetBankingPayment implements PaymentService
		
	The application only knows payment
	It doesn't care which implementation is used.
	
		
Abstract Class Definition

	An abstract class is a partially implemented class that provides both

		Common implementation
		Abstract methods for subclasses

	It cannot be instantiated.
	
	interface Notification
	
	EmailNotification
	SmsNotification
	PushNotification
	
	Why interface?

		Because each notification provider has completely different logic.
	

Interface: Contract

Abstract: Partial implementation

Interface: Cannot maintain object state

Abstract: Can maintain state


When do you choose Interface over Abstract Class?

Answer:

Choose an interface when:

	You want to define a contract.
	Classes are unrelated but should expose the same behavior.
	Multiple inheritance of type is required.
	Different implementations are expected.

Choose an abstract class when:

	Multiple subclasses share common code.
	You want to avoid duplicate logic.
	You need constructors or instance variables.
	You want to maintain state.

12+ Years Experience Answer

	In enterprise applications, I use interfaces to define contracts between layers because they promote loose coupling, 
	make dependency injection easier, and allow multiple implementations. Examples include service interfaces, repositories, 
	payment gateways, and strategy implementations.

	I use abstract classes when multiple implementations share significant common behavior or state. 
	This helps eliminate duplicate code while still allowing subclasses to customize specific functionality. 
	Common examples include template processing, report generation, employee hierarchies, and framework base classes.

Interview Follow-Up Questions
1. Can an interface have constructors?

No. Interfaces cannot have constructors because they cannot be instantiated.

2. Can an abstract class have constructors?

Yes. Constructors are used to initialize common fields when subclasses are created.

3. Can an interface have variables?

Yes, but they are always:

public static final

(constants).

4. Can an abstract class have instance variables?

Yes.

private String name;
protected int id;
5. Can a class implement multiple interfaces?

Yes.

class A implements X, Y, Z {}
6. Can a class extend multiple abstract classes?

No. Java does not support multiple class inheritance.

7. Why does Spring prefer interfaces?

Spring uses proxies for features like transactions, security, and AOP. Interfaces enable loose coupling and make it easier to swap implementations or mock dependencies in tests. Although modern Spring can proxy concrete classes (using CGLIB), defining interfaces is still a common design choice when multiple implementations are possible.

8. Which design patterns commonly use interfaces?
Strategy Pattern
Factory Pattern
Command Pattern
Observer Pattern
Dependency Injection

These patterns rely on interfaces to decouple clients from concrete implementations.

Key Interview Takeaway (12+ Years)

A senior engineer should emphasize design decisions, not just syntax:

Interface → Defines a contract, enables loose coupling, multiple implementations, dependency injection, and polymorphism.
Abstract Class → Provides a shared base implementation, common state, constructors, and reusable logic for closely related classes.

In real enterprise applications, it's common to see both used together: an interface defines the API (e.g., PaymentService), while an abstract base class (e.g., AbstractPaymentService) contains common validation, logging, or auditing logic that concrete implementations inherit. This approach balances flexibility with code reuse.

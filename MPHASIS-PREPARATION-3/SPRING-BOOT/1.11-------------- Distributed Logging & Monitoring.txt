# Distributed Logging & Monitoring

Distributed Logging and Monitoring are essential in **Microservices Architecture**, 
where multiple services run independently across different servers, containers, or cloud environments.

***

# Why Do We Need Distributed Logging & Monitoring?

Traditional Monolithic Application:

```text
Application
    ↓
Single Log File
```

Easy to debug.

***

Microservices Application:

```text
API Gateway
    ↓
User Service
    ↓
Order Service
    ↓
Payment Service
    ↓
Notification Service
```

Each service generates its own logs.

Challenges:

* Logs scattered across services
* Hard to trace requests
* Difficult root cause analysis
* Performance bottlenecks harder to identify

***

# Distributed Logging

Distributed Logging means collecting logs from multiple services into a centralized location.

## Flow

```text
Microservice A
      ↓
Microservice B
      ↓
Microservice C
      ↓
Log Collector
      ↓
Centralized Log Storage
      ↓
Dashboard/Search
```

# How to Correlate Logs with Traces?

In modern microservices, **log correlation** means linking logs from different services to the same request using a **Trace ID** (or Correlation ID).

Without correlation:

```text
Order Service Log
Payment Service Log
Inventory Service Log
Notification Service Log
```

You don't know which logs belong to the same request.

***

## Problem Example

User places an order.

```text
Client
  ↓
API Gateway
  ↓
Order Service
  ↓
Payment Service
  ↓
Inventory Service
```

If logs look like:

```text
Order Created
Payment Successful
Stock Updated
```

it's difficult to determine which request generated those logs.

***

## Solution: Trace ID

Generate a unique Trace ID at the entry point.

Example:

```text
TraceID = 7b8c9d123
```

Pass this ID to every service.

```text
Gateway
   ↓
TraceID=7b8c9d123

Order Service
   ↓
TraceID=7b8c9d123

Payment Service
   ↓
TraceID=7b8c9d123

Inventory Service
   ↓
TraceID=7b8c9d123
```

***

## Correlated Logs

```text
[TraceID=7b8c9d123]
Order Created

[TraceID=7b8c9d123]
Payment Successful

[TraceID=7b8c9d123]
Inventory Updated
```

Now searching by:

```text
7b8c9d123
```

shows the complete request journey.

***

# Trace ID vs Span ID

A distributed trace consists of:

```text
Trace
   ├── Span 1
   ├── Span 2
   ├── Span 3
```

### Trace ID

Represents the entire request.

```text
TraceID: abc123
```

### Span ID

Represents a single operation.

```text
Order Span
Payment Span
Inventory Span
```

Example:

```text
TraceID = abc123
SpanID  = span001
```

# Interview Answer

**How do you correlate logs with traces?**

> Generate a unique Trace ID at the start of a request, propagate it across all microservices, include the Trace ID in every log entry (typically via MDC/OpenTelemetry), and store logs and traces in centralized systems such as ELK and Jaeger. Using the common Trace ID, we can connect logs and distributed traces to view the complete end-to-end request flow.


---------------------------------------------------------------------


## 1. Add Trace ID to Logs Using MDC (SLF4J + Logback)

### Filter to Generate/Read Trace ID

```java
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import org.slf4j.MDC;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;
import java.util.UUID;

@Component
public class TraceIdFilter extends OncePerRequestFilter {

    public static final String TRACE_ID = "traceId";

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain)
            throws ServletException, IOException {

        String traceId = request.getHeader("X-Trace-Id");

        if (traceId == null || traceId.isBlank()) {
            traceId = UUID.randomUUID().toString();
        }

        MDC.put(TRACE_ID, traceId);

        response.setHeader("X-Trace-Id", traceId);

        try {
            filterChain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}
```

***

### Configure Logback

`logback-spring.xml`

```xml
<configuration>

    <appender name="CONSOLE"
              class="ch.qos.logback.core.ConsoleAppender">

        <encoder>
            <pattern>
                %d{yyyy-MM-dd HH:mm:ss}
                [%thread]
                %-5level
                [TraceId:%X{traceId}]
                %logger{36} - %msg%n
            </pattern>
        </encoder>

    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>

</configuration>
```

***

### Service Log

```java
@Service
public class OrderService {

    private static final Logger log =
            LoggerFactory.getLogger(OrderService.class);

    public void createOrder() {

        log.info("Creating Order");

        log.info("Order Created Successfully");
    }
}
```

***

### Output

```text
2026-07-06 10:20:15
INFO
[TraceId:8c9b1234-a12b-45cd]
OrderService - Creating Order

2026-07-06 10:20:15
INFO
[TraceId:8c9b1234-a12b-45cd]
OrderService - Order Created Successfully
```

All logs for the request contain the same Trace ID.

# 2. Propagate Trace ID in HTTP Headers

Suppose:

```text
API Gateway
    ↓
Order Service
    ↓
Payment Service
```

The same Trace ID must travel through all HTTP calls.

***

## Incoming Header

```http
GET /orders

X-Trace-Id: abc123
```

***

## Pass Header Using RestTemplate

### Configuration

```java
@Bean
public RestTemplate restTemplate() {
    return new RestTemplate();
}
```

***

### Propagate Trace ID

```java
import org.slf4j.MDC;
import org.springframework.http.*;
import org.springframework.web.client.RestTemplate;

@Service
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;

    public void processOrder() {

        String traceId = MDC.get("traceId");

        HttpHeaders headers = new HttpHeaders();
        headers.set("X-Trace-Id", traceId);

        HttpEntity<String> entity =
                new HttpEntity<>(headers);

        restTemplate.exchange(
                "http://payment-service/pay",
                HttpMethod.POST,
                entity,
                String.class);
    }
}
```

***

## Receiving Service

```java
@RestController
public class PaymentController {

    @PostMapping("/pay")
    public String pay(
            @RequestHeader("X-Trace-Id")
            String traceId) {

        MDC.put("traceId", traceId);

        log.info("Payment Started");

        return "SUCCESS";
    }
}
```

## Interview Answer

**How do you add Trace ID to logs?**

> Store the Trace ID in MDC (`MDC.put("traceId", traceId)`), configure Logback to print `%X{traceId}`, and clear MDC after request completion.

**How do you propagate Trace ID in HTTP headers?**

> Read or generate the Trace ID at the gateway/service entry point, add it to headers such as `X-Trace-Id`, and forward the same header in all downstream service calls using RestTemplate, WebClient, Feign, or OpenTelemetry.

# How to Propagate Trace ID in Kafka?

Unlike HTTP, Kafka does not have request headers like `X-Trace-Id`. Instead, Kafka messages support **record headers**, which are the recommended way to propagate Trace IDs between producers and consumers.

***

## Architecture

```text
Order Service
     │
     │ TraceID=abc123
     ▼
Kafka Producer
     ▼
Kafka Topic
     ▼
Kafka Consumer
     ▼
Payment Service

TraceID=abc123
```

The same Trace ID travels with the Kafka message.

***

# 1. Kafka Producer: Add Trace ID to Message Header

Suppose the Trace ID is stored in MDC.

```java
String traceId = MDC.get("traceId");
```

Create a Kafka record with headers:

```java
import org.apache.kafka.clients.producer.ProducerRecord;

ProducerRecord<String, String> record =
        new ProducerRecord<>(
                "order-topic",
                "Order Created");

record.headers().add(
        "traceId",
        traceId.getBytes());

kafkaTemplate.send(record);
```

***

## Producer Example

```java
@Service
public class OrderProducer {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void publishOrder() {

        String traceId = MDC.get("traceId");

        ProducerRecord<String, String> record =
                new ProducerRecord<>(
                        "order-topic",
                        "Order Created");

        record.headers().add(
                "traceId",
                traceId.getBytes());

        kafkaTemplate.send(record);
    }
}
```

***

# 2. Kafka Consumer: Read Trace ID from Header

Consumer receives the header and restores it into MDC.

```java
@KafkaListener(topics = "order-topic")
public void consume(
        ConsumerRecord<String, String> record) {

    Header header =
            record.headers().lastHeader("traceId");

    String traceId =
            new String(header.value());

    MDC.put("traceId", traceId);

    try {
        log.info("Processing Order");
    } finally {
        MDC.clear();
    }
}
```

***

## Output

Producer logs:

```text
[TraceId:abc123]
Order Published
```

Consumer logs:

```text
[TraceId:abc123]
Processing Order
```

Now logs from both services can be correlated.

***

# 3. Generic Utility Method

To avoid duplicate code:

### Header Utility

```java
public class TraceUtil {

    public static final String TRACE_ID =
            "traceId";

    public static String getTraceId(
            ConsumerRecord<?, ?> record) {

        Header header =
            record.headers()
                  .lastHeader(TRACE_ID);

        if(header == null) {
            return null;
        }

        return new String(header.value());
    }
}
```

Consumer:

```java
String traceId =
        TraceUtil.getTraceId(record);

MDC.put("traceId", traceId);
```

# 5. Using OpenTelemetry with Kafka (Recommended)

Modern systems typically use **OpenTelemetry**.

OpenTelemetry automatically:

* Creates Trace IDs
* Creates Span IDs
* Injects tracing context into Kafka headers
* Extracts tracing context on the consumer side

Producer:

```java
Span.current();
```

Consumer:

```java
Span.current();
```

OpenTelemetry handles propagation automatically.

Architecture:

```text
Service A
    ▼
OpenTelemetry
    ▼
Kafka Headers
    ▼
OpenTelemetry
    ▼
Service B
```

You don't need to manually add:

```text
traceId
spanId
```

to headers.

***

# Spring Kafka Interceptor Example

### Producer Interceptor

```java
public class TraceProducerInterceptor
        implements ProducerInterceptor<String, String> {

    @Override
    public ProducerRecord<String, String> onSend(
            ProducerRecord<String, String> record) {

        String traceId = MDC.get("traceId");

        if(traceId != null) {
            record.headers().add(
                    "traceId",
                    traceId.getBytes());
        }

        return record;
    }
}
```

This automatically adds Trace IDs to every message.

In production systems, OpenTelemetry is commonly used to automatically propagate tracing context through Kafka headers and correlate logs, metrics, and traces across services.

**Interview Answer (12+ Years):**

> There are **21 ways** to handle circular dependency in Spring Boot, but the preferred approaches are **redesigning the architecture, 
extracting common functionality, using constructor injection, event-driven communication, and applying design patterns like Facade or
 Mediator.** Temporary solutions include `@Lazy`, `ObjectProvider`, and enabling circular references, which should generally be avoided.

1. **Redesign the application architecture**


	@Service
	public class OrderService {
		public void createOrder() {
			// order logic
		}
	}

	@Service
	public class PaymentService {
		public void processPayment() {
			// payment logic
		}
	}

	@Service
	public class CheckoutService {

		private final OrderService orderService;
		private final PaymentService paymentService;

		public CheckoutService(OrderService orderService,
							   PaymentService paymentService) {
			this.orderService = orderService;
			this.paymentService = paymentService;
		}

		public void checkout() {
			orderService.createOrder();
			paymentService.processPayment();
		}
	}
	
	
**Extract common logic into a third service**
--------------------------------------------------------------------
	
	After (Extract Common Logic into a Third Service ✅)

Suppose both services need some common functionality. Move that logic to CommonService.
		
	@Service
	public class CommonService {

		public void commonLogic() {
			System.out.println("Common logic");
		}
	}
		
	@Service
	public class BService {

		private final CommonService commonService;

		public BService(CommonService commonService) {
			this.commonService = commonService;
		}

		public void methodB() {
			commonService.commonLogic();
		}
	}

	@Service
	public class AService {

		private final CommonService commonService;

		public AService(CommonService commonService) {
			this.commonService = commonService;
		}

		public void methodA() {
			commonService.commonLogic();
		}
	}
	Dependency Graph
			CommonService
			↑          ↑
		AService    BService

	No cycle exists.

	Why it works?
		AService no longer depends on BService.
		BService no longer depends on AService.
		Shared functionality is centralized in CommonService.
		This improves maintainability and follows the Single Responsibility Principle (SRP).

	This is one of the best and most maintainable ways to eliminate circular dependencies in Spring Boot.
	
	
	**Apply Single Responsibility Principle (SRP)**
	
Applying the Single Responsibility Principle (SRP) means each class should have only one reason to change. 
By separating responsibilities, you can eliminate circular dependencies.

Before (Violates SRP ❌)


@Service
public class OrderService {

    @Autowired
    private PaymentService paymentService;

    public void createOrder() {
        // order logic
        paymentService.processPayment();
    }
}

@Service
public class PaymentService {

    @Autowired
    private OrderService orderService;

    public void processPayment() {
        // payment logic
        orderService.updateOrderStatus();
    }
}

OrderService → PaymentService → OrderService ❌

After (Apply SRP ✅)

Separate responsibilities:


		@Service
		public class OrderService {

			public void createOrder() {
				// order logic
			}

			public void updateOrderStatus() {
				// status update logic
			}
		}

		@Service
		public class PaymentService {

			public void processPayment() {
				// payment logic
			}
		}


		@Service
		public class CheckoutService {

			private final OrderService orderService;
			private final PaymentService paymentService;

			public CheckoutService(OrderService orderService,
								   PaymentService paymentService) {
				this.orderService = orderService;
				this.paymentService = paymentService;
			}

			public void checkout() {
				orderService.createOrder();
				paymentService.processPayment();
				orderService.updateOrderStatus();
			}
		}

Dependency Graph
        CheckoutService
         /          \
OrderService    PaymentService

No cycle exists.

Simple definition of SRP

	A class should have only one responsibility and one reason to change.

Applying SRP often removes circular dependencies naturally by separating concerns into dedicated services.

**Use ObjectProvider<T>**

Yes, ObjectProvider<T> can be used to break a circular dependency by obtaining the dependency lazily (on demand) 
instead of during bean creation.

	@Service
	public class AService {

		private final ObjectProvider<BService> bServiceProvider;

		public AService(ObjectProvider<BService> bServiceProvider) {
			this.bServiceProvider = bServiceProvider;
		}

		public void doSomething() {
			BService bService = bServiceProvider.getObject();
			bService.doSomethingElse();
		}
	}

	@Service
	public class BService {

		private final AService aService;

		public BService(AService aService) {
			this.aService = aService;
		}

		public void doSomethingElse() {
			System.out.println("BService");
		}
	}

How it works:

	Spring creates AService and injects an ObjectProvider<BService>.
	Spring creates BService and injects AService.
	BService is not requested from the provider until doSomething() is called.
		
	AService → ObjectProvider<BService>
	BService → AService

	Thus, the circular dependency is avoided.

		Note: ObjectProvider<T> is a workaround for lazy access to beans. 
		The preferred solution is still to redesign the application architecture to remove the circular dependency.

Yes, ApplicationContext.getBean() can be used to break a circular dependency by retrieving the dependent bean 
only when it is actually needed.

	@Service
	public class AService {

		@Autowired
		private ApplicationContext context;

		public void doSomething() {
			BService bService = context.getBean(BService.class);
			bService.doSomethingElse();
		}
	}

	@Service
	public class BService {

		@Autowired
		private AService aService;

		public void doSomethingElse() {
			System.out.println("BService");
		}
	}
	
How it works:

	Spring creates AService and injects ApplicationContext.
	Spring creates BService and injects AService.
	AService does not depend directly on BService.
	
	When doSomething() is called, AService obtains BService from the Spring container:	
	
	BService bService = context.getBean(BService.class);
	
Dependency Graph:
	
	AService → ApplicationContext
	BService → AService

	At runtime:
	AService → BService
	
Problem with this approach

	❌ Tightly couples your code to Spring.
	❌ Violates Dependency Injection and Inversion of Control principles.
	❌ Makes unit testing harder.
	❌ Hides dependencies.
	⚠️ Considered a workaround, not a clean design.	
	
Simple answer

Yes, ApplicationContext.getBean() can resolve circular dependencies by performing a lazy lookup,
 but it is generally not recommended. Redesigning the application architecture is preferred.

**Use Asynchronous Messaging (Kafka, RabbitMQ, ActiveMQ)** i need kafka only

You can eliminate a circular dependency by using Kafka and communicating through events instead of direct service calls.

Before (Circular Dependency ❌)
OrderService → PaymentService
PaymentService → OrderService

After (Kafka Event-Driven Architecture ✅)

OrderService --(OrderCreatedEvent)--> Kafka Topic
                                        ↓
                                  PaymentService

PaymentService --(PaymentCompletedEvent)--> Kafka Topic
                                             ↓
                                       OrderService
									   
Neither service depends on the other.

1. OrderCreatedEvent

public class OrderCreatedEvent {
    private Long orderId;

    public OrderCreatedEvent() {}

    public OrderCreatedEvent(Long orderId) {
        this.orderId = orderId;
    }

    public Long getOrderId() {
        return orderId;
    }

    public void setOrderId(Long orderId) {
        this.orderId = orderId;
    }
}

2. OrderService (Producer)


@Service
public class OrderService {

    private final KafkaTemplate<String, OrderCreatedEvent> kafkaTemplate;

    public OrderService(KafkaTemplate<String, OrderCreatedEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void createOrder(Long orderId) {

        // order logic

        kafkaTemplate.send(
            "order-created-topic",
            new OrderCreatedEvent(orderId)
        );
    }
}

3. PaymentService (Consumer)

@Service
public class PaymentService {

    @KafkaListener(
        topics = "order-created-topic",
        groupId = "payment-group"
    )
    public void processPayment(OrderCreatedEvent event) {

        System.out.println(
            "Processing payment for order " + event.getOrderId()
        );

        // payment logic
    }
}

Dependency Graph

OrderService
      |
      | (publish event)
      v
     Kafka
      ^
      | (consume event)
	  
PaymentService

No direct dependency exists:

OrderService ↔ PaymentService ❌

OrderService → Kafka → PaymentService ✅

Using Kafka is a good solution when services should be loosely coupled and communicate asynchronously, 
completely avoiding circular dependencies.

Instead of having services call each other directly:


21. **Enable circular references using `spring.main.allow-circular-references=true` (Not Recommended)**


Yes, Spring Boot allows you to enable circular references, but **this is not recommended** because it hides design problems.

### Step 1: Enable Circular References

In `application.properties`:

```properties
spring.main.allow-circular-references=true
```

Or in `application.yml`:

```yaml
spring:
  main:
    allow-circular-references: true
```

### Step 2: Example

```java
@Service
public class AService {

    @Autowired
    private BService bService;

    public void methodA() {
        System.out.println("AService");
    }
}
```

```java
@Service
public class BService {

    @Autowired
    private AService aService;

    public void methodB() {
        System.out.println("BService");
    }
}
```

```text
AService ↔ BService
```

With:

```properties
spring.main.allow-circular-references=true
```

Spring will try to resolve the cycle by exposing an early bean reference.

### Why is this not recommended?

* ❌ Hides poor design.
* ❌ Can lead to partially initialized beans.
* ❌ Makes code harder to understand and maintain.
* ❌ Does not work for constructor-based circular dependencies.
* ❌ Future versions of Spring may further restrict this behavior.

### Simple answer

```properties
spring.main.allow-circular-references=true
```

**allows Spring to resolve some circular dependencies (typically field/setter injection), but it is a workaround,
 not a proper fix. Redesigning the application is the preferred solution.**

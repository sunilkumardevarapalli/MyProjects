WebClient is a reactive HTTP client introduced in Spring WebFlux. By default, 
it supports asynchronous and non-blocking communication using Mono and Flux. However, 
if .block() is used, it can also work synchronously. Therefore, WebClient supports both synchronous and asynchronous communication, 
but it is primarily intended for asynchronous communication.

Service-to-Service Communication in Microservices (Spring Boot)

In a microservices architecture, services often need to communicate with each other to fulfill business requirements. In Spring Boot, this communication can be implemented using synchronous or asynchronous approaches.

1. Synchronous Communication

The calling service waits for a response from another service.

a) REST API Communication

Most common approach using HTTP.

Using RestTemplate (Legacy)
@RestController
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;

    public String getCustomer(Long customerId) {
        String url = "http://customer-service/customers/" + customerId;
        return restTemplate.getForObject(url, String.class);
    }
}


Configuration:

@Bean
public RestTemplate restTemplate() {
    return new RestTemplate();
}

Using WebClient (Recommended)

Non-blocking and reactive.

@Service
public class OrderService {

    private final WebClient webClient;

    public OrderService(WebClient.Builder builder) {
        this.webClient = builder.baseUrl("http://customer-service").build();
    }

    public Mono<Customer> getCustomer(Long customerId) {
        return webClient.get()
                .uri("/customers/{id}", customerId)
                .retrieve()
                .bodyToMono(Customer.class);
    }
}

b) OpenFeign Client

Simplifies REST calls.

Dependency
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>


Enable Feign:

@SpringBootApplication
@EnableFeignClients
public class OrderApplication {
}


Feign Client:

@FeignClient(name = "customer-service")
public interface CustomerClient {

    @GetMapping("/customers/{id}")
    Customer getCustomer(@PathVariable Long id);
}


Usage:

@Service
public class OrderService {

    @Autowired
    private CustomerClient customerClient;

    public Customer getCustomer(Long id) {
        return customerClient.getCustomer(id);
    }
}

2. Service Discovery

Instead of hardcoding URLs, use a service registry.

Eureka Server
<dependency>

Application properties:

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/


Services register themselves and communicate using service names.

Example:

http://customer-service/customers/1

3. Asynchronous Communication

Services communicate via messages/events.

Apache Kafka

Order Service → Kafka Topic → Inventory Service

Producer:

@Autowired
private KafkaTemplate<String, String> kafkaTemplate;

public void publishOrder(String order) {
    kafkaTemplate.send("order-topic", order);
}


Consumer:

@KafkaListener(topics = "order-topic")
public void consume(String message) {
    System.out.println("Received: " + message);
}

No blocking calls

For Spring Boot microservices, Resilience4j is the recommended library for implementing Retry, Circuit Breaker, Bulkhead, Rate Limiter, and Timeouts.

2. Retry
Use Case

If Customer Service is temporarily unavailable, retry 3 times before failing.

Code
@Service
@RequiredArgsConstructor
public class CustomerService {

    private final CustomerClient customerClient;

    @Retry(name = "customerRetry", fallbackMethod = "fallbackCustomer")
    public Customer getCustomer(Long id) {
        return customerClient.getCustomer(id);
    }

    public Customer fallbackCustomer(Long id, Exception ex) {
        return new Customer(id, "Fallback Customer");
    }
}

Configuration
resilience4j:
  retry:
    instances:
      customerRetry:
        max-attempts: 3
        wait-duration: 2s
        retry-exceptions:
          - java.net.ConnectException
          - feign.RetryableException

Flow
Request
   |
Fail
   |
Retry #1
   |
Retry #2
   |
Retry #3
   |
Fallback

3. Circuit Breaker
Use Case

Prevent continuous calls to a failing service.

Code
@CircuitBreaker(
        name = "customerCB",
        fallbackMethod = "fallbackCustomer")
public Customer getCustomer(Long id) {

    return customerClient.getCustomer(id);
}

public Customer fallbackCustomer(Long id, Exception ex) {
    return new Customer(id, "Default Customer");
}

Configuration
resilience4j:
  circuitbreaker:
    instances:
      customerCB:
        sliding-window-size: 10
        minimum-number-of-calls: 5
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10s
        permitted-number-of-calls-in-half-open-state: 3

States
CLOSED
   |
50% failures
   |
OPEN
   |
10 seconds wait
   |
HALF_OPEN
   |
Success -> CLOSED
Failure -> OPEN

4. Rate Limiter
Use Case

Allow only 5 requests per second.

Code
@RateLimiter(
        name = "customerRateLimiter",
        fallbackMethod = "rateLimitFallback")
public Customer getCustomer(Long id) {

    return customerClient.getCustomer(id);
}

public Customer rateLimitFallback(Long id, Exception ex) {

    throw new RuntimeException(
            "Too many requests. Please try later.");
}

Configuration
resilience4j:
  ratelimiter:
    instances:
      customerRateLimiter:
        limit-for-period: 5
        limit-refresh-period: 1s
        timeout-duration: 0

Flow
1 sec window

Req1 ✅
Req2 ✅
Req3 ✅
Req4 ✅
Req5 ✅
Req6 ❌ Rejected
Req7 ❌ Rejected

5. Bulkhead
Use Case

Protect service resources from being exhausted.

Suppose only 10 threads can call Customer Service simultaneously.

Code
@Bulkhead(
        name = "customerBulkhead",
        type = Bulkhead.Type.SEMAPHORE,
        fallbackMethod = "bulkheadFallback")
public Customer getCustomer(Long id) {

    return customerClient.getCustomer(id);
}

public Customer bulkheadFallback(Long id, Exception ex) {

    return new Customer(id, "Bulkhead Fallback Customer");
}

Configuration
resilience4j:
  bulkhead:
    instances:
      customerBulkhead:
        max-concurrent-calls: 10
        max-wait-duration: 2s

Flow
Incoming Requests = 50

10 allowed ✅
40 blocked ❌

ThreadPool Bulkhead

Useful for async operations.

@Bulkhead(
     name = "customerThreadPool",
     type = Bulkhead.Type.THREADPOOL)
public CompletableFuture<Customer> getCustomer(Long id) {

    return CompletableFuture.supplyAsync(
              () -> customerClient.getCustomer(id));
}


Configuration:

resilience4j:
  thread-pool-bulkhead:
    instances:
      customerThreadPool:
        max-thread-pool-size: 10
        core-thread-pool-size: 5
        queue-capacity: 20

6. Timeout Handling
Use Case

Customer Service should respond within 3 seconds.

Code
@TimeLimiter(
        name = "customerTimeout")
public CompletableFuture<Customer> getCustomer(Long id) {

    return CompletableFuture.supplyAsync(
            () -> customerClient.getCustomer(id));
}

Configuration
resilience4j:
  timelimiter:
    instances:
      customerTimeout:
        timeout-duration: 3s
        cancel-running-future: true

Timeout Fallback
@TimeLimiter(name = "customerTimeout")
@CircuitBreaker(name = "customerCB",
                fallbackMethod = "timeoutFallback")
public CompletableFuture<Customer> getCustomer(Long id) {

    return CompletableFuture.supplyAsync(
        () -> customerClient.getCustomer(id));
}

public CompletableFuture<Customer> timeoutFallback(
        Long id,
        Exception ex) {

    return CompletableFuture.completedFuture(
            new Customer(id, "Timeout Fallback"));
}

Combining All Patterns

This is a real-world implementation:

@Service
@RequiredArgsConstructor
public class CustomerService {

    private final CustomerClient customerClient;

    @Retry(name = "customerRetry")
    @CircuitBreaker(
            name = "customerCB",
            fallbackMethod = "fallback")
    @RateLimiter(name = "customerRateLimiter")
    @Bulkhead(name = "customerBulkhead")
    public Customer getCustomer(Long id) {

        return customerClient.getCustomer(id);
    }

    public Customer fallback(Long id, Exception ex) {

        return new Customer(id,
                "Fallback Customer");
    }
}

      customerCB:
        sliding-window-size: 10
        failure-rate-threshold: 50
        minimum-number-of-calls: 5
        wait-duration-in-open-state: 10s

  ratelimiter:
    instances:
      customerRateLimiter:
        limit-for-period: 5
        limit-refresh-period: 1s
        timeout-duration: 0

  bulkhead:
    instances:
      customerBulkhead:
        max-concurrent-calls: 10
        max-wait-duration: 2s

  timelimiter:
    instances:
      customerTimeout:
        timeout-duration: 3s
        cancel-running-future: true

Production Best Practice Order
@Retry
@CircuitBreaker
@RateLimiter
@Bulkhead
@TimeLimiter


This order ensures retries happen first, failing services are isolated by the circuit breaker, request throttling is applied, resource consumption is controlled, and long-running calls are timed out.

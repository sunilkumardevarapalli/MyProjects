# Java Class Loading, ClassLoader Leaks & Best Practices

## CLASSPATH

**Classpath** is the location where the JVM searches for `.class` files and JARs required by the application.

### Class Loading Failures

#### 1. ClassNotFoundException

Occurs when the JVM cannot find the specified class while loading it dynamically.


Class.forName("com.example.Employee");


**Reason:** Class is not available in the classpath.

***

#### 2. NoClassDefFoundError

Occurs when a class was available during compilation but is missing at runtime.


java.lang.NoClassDefFoundError


**Reason:** The required class or dependency is not present during execution.

***

#### 3. VerifyError

Occurs when JVM bytecode verification fails.


java.lang.VerifyError


**Reason:** Invalid or corrupted bytecode.

***

#### 4. UnsupportedClassVersionError

Occurs when a class is compiled with a newer Java version than the JVM running it.


java.lang.UnsupportedClassVersionError


***

#### 5. ExceptionInInitializerError

Occurs when an exception is thrown during static initialization.


public class Employee {

    static {
        int x = 10 / 0;
    }
}


**Output**


ExceptionInInitializerError


***

# Custom Class Loader

A **Custom Class Loader** is created by extending the `ClassLoader` class.

It allows loading classes from sources other than the standard classpath:

* Database
* Network
* Encrypted Files
* Cloud Storage
* Plugin Directories

### Important Methods


loadClass()

# Memory Leaks Related to Class Loaders

ClassLoader leaks are common in enterprise applications and application servers, especially during redeployments.

## Common Causes

### 1. Static Variable Leaks


private static List<Object> cache = new ArrayList<>();


Static objects retain references and prevent class unloading.

***

### 2. ThreadLocal Leaks


threadLocal.remove();


Objects stored in `ThreadLocal` remain attached to threads if not removed.

***

### 3. Running Threads

Background threads started by an application may continue running after deployment.


executorService.shutdown();


***

### 4. JDBC Driver Leaks

JDBC drivers registered with `DriverManager` can keep references to application class loaders.

***

### 5. Cached Objects

Objects stored in caches may retain references to loaded classes.

***

# Preventing ClassLoader Leaks

✅ Clear static collections and caches

✅ Remove ThreadLocal values


threadLocal.remove();


✅ Stop background threads


executorService.shutdown();


✅ Deregister JDBC drivers


DriverManager.deregisterDriver(driver);


✅ Close connection pools


hikariDataSource.close();


✅ Avoid long-lived static references

✅ Reuse class loaders where appropriate

✅ Use dependency injection instead of global static state

✅ Use `WeakReference` when suitable

✅ Monitor Metaspace regularly

***

# JDBC & HikariCP Cleanup

Always close the connection pool during application shutdown:


hikariDataSource.close();


This releases:

* Database connections
* Worker threads
* Internal caches
* ClassLoader references

***

# Performance Best Practices

### Improve Class Loading Performance

✅ Load classes lazily whenever possible

✅ Reuse existing class loaders

✅ Avoid creating excessive custom class loaders

✅ Reduce unnecessary dynamic class loading

✅ Avoid excessive reflection

✅ Use Class Data Sharing (CDS)

✅ Minimize heavy static initialization blocks

✅ Monitor Metaspace usage

✅ Prevent ClassLoader leaks

## 1. How to Use `WeakReference` Effectively?

A `WeakReference` allows an object to be garbage collected when no strong references exist.

### Why Use It?

Normally:


Employee emp = new Employee();


`emp` is a **strong reference**, so the object cannot be garbage collected.

With `WeakReference`:


Employee emp = new Employee();

WeakReference<Employee> ref =
        new WeakReference<>(emp);

emp = null;   // remove strong reference


Now the object becomes eligible for garbage collection.

***

### Common Use Cases

#### 1. Memory-Sensitive Caching

Instead of:


Map<String, Employee> cache = new HashMap<>();


Use:


Map<String, WeakReference<Employee>> cache =
        new HashMap<>();


When memory becomes scarce, JVM can automatically remove cached objects.

***

#### 2. Preventing ClassLoader Leaks

Suppose a static cache stores objects loaded by an application's ClassLoader:


private static List<Object> cache = new ArrayList<>();


The static reference keeps the objects alive and prevents class unloading.

Using:


private static List<WeakReference<Object>> cache =
        new ArrayList<>();


reduces the chance of retaining objects unnecessarily.

***

### Important Limitation

Never rely on a `WeakReference` if the object is essential.


Employee emp = ref.get();

if(emp != null) {
    emp.process();
}


The object may disappear at any garbage collection cycle.

***

### Interview Answer

**WeakReference is used for memory-sensitive objects such as caches, listeners, and metadata.
 It allows the garbage collector to reclaim memory when no strong references exist,
 helping reduce memory leaks and ClassLoader retention issues.**

***

# 2. How Does CDS (Class Data Sharing) Improve Performance?

### What is CDS?

**Class Data Sharing (CDS)** is a JVM feature that stores preprocessed class metadata in a shared archive file.

Instead of every JVM:

1. Loading classes
2. Verifying bytecode
3. Linking classes
4. Creating metadata

from scratch, it can reuse the shared archive.

***

### Without CDS

Every JVM startup performs:


Load Class
   ↓
Verify
   ↓
Prepare
   ↓
Resolve
   ↓
Initialize


for thousands of classes.

This increases startup time and memory usage.

***

### With CDS


Shared Archive
      ↓
Mapped directly into memory
      ↓
Faster startup
      ↓
Less memory usage


Multiple JVMs can share class metadata.

***

### Benefits

#### Faster Startup

Classes are already processed and archived.


Application starts faster


***

#### Lower Memory Consumption

Common JDK classes can be shared across JVM processes.


JVM A
JVM B
JVM C
   ↓
Shared Metadata


Less duplicated memory.

***

#### Better Container Performance

CDS is extremely useful for:

* Spring Boot Applications
* Microservices
* Docker Containers
* Kubernetes Deployments

where fast startup matters.

***

### Interview Answer

**CDS improves JVM startup performance by storing preprocessed class metadata in a shared archive and reusing it across JVM instances. This reduces class loading overhead, lowers memory consumption, and speeds application startup.**

***

# 3. What is the Role of `defineClass()`?

### Definition

`defineClass()` is the core method used by a ClassLoader to convert raw bytecode into a Java `Class` object.

***

### Class Loading Flow


.class file
      ↓
Read bytecode
      ↓
defineClass()
      ↓
Class object created
      ↓
resolveClass()
      ↓
Ready for execution


***

### Example


public class MyClassLoader extends ClassLoader {

    @Override
    protected Class<?> findClass(String name)
            throws ClassNotFoundException {

        byte[] bytes = loadByteCode(name);

        return defineClass(
                name,
                bytes,
                0,
                bytes.length
        );
    }
}


***

### What Happens Inside `defineClass()`?


Byte Array
    ↓
Verify Bytecode
    ↓
Create Class Metadata
    ↓
Store in Metaspace
    ↓
Return Class Object


Example:


byte[] classBytes = ...;

Class<?> clazz =
     defineClass(
         "Employee",
         classBytes,
         0,
         classBytes.length
     );


***

### Why Is It Important?

Every custom ClassLoader eventually relies on:


defineClass()


to create the actual Java class.

Without it:


Bytecode exists
But no Class object exists


and JVM cannot instantiate or execute the class.

***

### Interview Answer

**`defineClass()` is the fundamental ClassLoader method that converts raw class bytecode into a JVM Class object. It performs bytecode verification, creates class metadata in Metaspace, and makes the class available for use by the application.**

***

| Concept         | Purpose                                                                 |
| --------------- | ----------------------------------------------------------------------- |
| `WeakReference` | Allows GC to reclaim objects when only weak references remain           |
| CDS             | Shares preloaded class metadata to reduce startup time and memory usage |
| `defineClass()` | Converts bytecode into a JVM `Class` object during class loading        |

## Does CDS Work Only When Multiple JVMs Exist?

**No.** CDS provides benefits even with a **single JVM** because class metadata is already preprocessed and stored in an archive, 
reducing class loading and verification work during startup. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)


Single JVM
    ↓
Reads preprocessed classes from CDS archive
    ↓
Faster startup


The **additional memory-sharing benefit** appears when multiple JVM processes on the same machine use the same archive. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[openjdk.org\]](https://openjdk.org/jeps/310)


JVM A
JVM B
JVM C
   ↓
Shared CDS Archive
   ↓
Reduced total memory usage


Think of it as:

* **1 JVM** → Faster startup. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)
* **Multiple JVMs** → Faster startup + reduced memory footprint. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[openjdk.org\]](https://openjdk.org/jeps/310)

***

## Create Your Own CDS Archive (AppCDS)

For application classes (Spring Boot, Hibernate, etc.), create a custom archive.

### Step 1: Generate Class List

bash
java \
-XX:DumpLoadedClassList=classes.lst \
-cp app.jar \
com.example.Main
`

### Step 2: Generate Archive

bash
java \
-Xshare:dump \
-XX:SharedClassListFile=classes.lst \
-XX:SharedArchiveFile=app.jsa \
-cp app.jar


### Step 3: Use Archive

bash
java \
-Xshare:on \
-XX:SharedArchiveFile=app.jsa \
-cp app.jar \
com.example.Main


This is called **Application CDS (AppCDS)**. [\[openjdk.org\]](https://openjdk.org/jeps/310), [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html)

***

## Interview Answer

**How does CDS work if there is only one JVM?**

> Even with a single JVM, CDS improves startup time because core class metadata is preprocessed and loaded from a shared archive. 
When multiple JVMs use the same archive, they can additionally share read-only class metadata, reducing overall memory consumption on
 the machine. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[openjdk.org\]](https://openjdk.org/jeps/310)

That's usually the answer interviewers expect.

## How to Verify if CDS is Active?

### 1. Check CDS Logs (Recommended)

Run:

bash
java -Xlog:cds -version


If CDS is active, you'll see messages similar to:


Opened archive ...
Using shared archive ...


This confirms the JVM loaded the CDS archive. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)

***

### 2. Print VM Settings

bash
java -XshowSettings:vm -version


Look for CDS-related information in the VM settings output. Modern JDKs ship with a default CDS archive enabled by default. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)

***

### 3. Explicitly Enable CDS and Verify

bash
java -Xshare:on -Xlog:cds -version


Expected output:


Using shared archive


If the archive cannot be used, the JVM reports the reason. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)

***

### 4. Check if CDS Is Disabled

bash
java -Xshare:off -Xlog:cds -version


Output will indicate CDS is disabled. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html)

***

### 5. Verify AppCDS Archive

For a custom archive:

bash
java \
-Xshare:on \
-XX:SharedArchiveFile=app.jsa \
-Xlog:cds \
-jar app.jar


Look for:


Opened archive app.jsa
Using shared archive


This confirms your AppCDS archive is being used. [\[openjdk.org\]](https://openjdk.org/jeps/310), [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html)

***

## Interview Answer

**How do you verify whether CDS is active?**

> Use JVM logging:
>
> bash
> java -Xlog:cds -version
> 
>
> If the output contains messages such as **"Using shared archive"** or **"Opened archive"**, CDS is active and the JVM is loading classes from the shared archive instead of processing them from scratch. [\[docs.oracle.com\]](https://docs.oracle.com/en/java/javase/26/vm/class-data-sharing.html), [\[dev.java\]](https://dev.java/learn/jvm/cds-appcds/)

bash

✅ `"Using shared archive"` → CDS enabled

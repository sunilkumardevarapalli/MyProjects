Interview Answer

@PostConstruct, afterPropertiesSet(), and initMethod are all bean initialization callbacks executed after dependency injection. 

@PostConstruct is a standard annotation and is preferred for application-level initialization. 

afterPropertiesSet() comes from the InitializingBean interface and is used mainly in framework-related code but 
couples the class to Spring. 

initMethod is configured externally and is useful for initializing third-party or legacy classes that cannot be modified. 

Their invocation order is:

@PostConstruct → afterPropertiesSet() → initMethod.

# 1. Initializing Cache

Suppose you want to load all active employees into an in-memory cache when the application starts.


@Service
public class EmployeeService {

    @Autowired
    private EmployeeRepository employeeRepository;

    private Map<Long, Employee> employeeCache = new ConcurrentHashMap<>();

    @PostConstruct
    public void loadCache() {
        List<Employee> employees = employeeRepository.findAll();

        employees.forEach(emp ->
            employeeCache.put(emp.getId(), emp));

        System.out.println("Employee cache initialized");
    }

    public Employee getEmployee(Long id) {
        return employeeCache.get(id);
    }
}


### Executed:

* Once during application startup.

# 2. Loading Configuration

Suppose configuration values come from `application.properties`.

properties
app.max.connections=100
app.timeout=5000



@Service
public class AppConfigService {

    @Value("${app.max.connections}")
    private int maxConnections;

    @Value("${app.timeout}")
    private int timeout;

    @PostConstruct
    public void init() {
        System.out.println("Max Connections : " + maxConnections);
        System.out.println("Timeout : " + timeout);
    }
}


### Why `@PostConstruct`?

At constructor time, `@Value` fields are not injected yet.

# 3. Validating Injected Dependencies

Suppose a datasource must be present.


@Service
public class UserService {

    @Autowired
    private DataSource dataSource;

    @PostConstruct
    public void validate() {

        if (dataSource == null) {
            throw new IllegalStateException("Datasource is missing");
        }

        System.out.println("Datasource validated");
    }
}


Another example:


@Service

        if (smtpHost == null || smtpHost.isBlank()) {
            throw new RuntimeException("SMTP Host not configured");
        }
    }
}


### Benefit

Application fails fast during startup instead of failing later.

1. **Initializing cache**

   
   cache.putAll(repository.findAll());
   

2. **Loading configuration values**

   countries = repository.findAll();
   

`@PostConstruct` runs once after dependency injection and before the bean is made available for use, making it the preferred mechanism for application-level initialization.

when we go for @PreDestroy / destroy() / destroy-method real time examples

`@PreDestroy`, `DisposableBean.destroy()`, and `destroyMethod` are used for **cleanup activities** when the Spring container is shutting down. They are analogous to `@PostConstruct`, but for destruction.

The invocation order is:


@PreDestroy
    ↓
DisposableBean.destroy()
    ↓

## 1. Closing Database Connections


@Service
public class DatabaseService {

    private Connection connection;

    @PostConstruct
    public void init() throws Exception {
        connection = DriverManager.getConnection(url, user, password);
    }

    @PreDestroy
    public void closeConnection() throws Exception {
        if (connection != null) {
            connection.close();
        }
    }
}

# 1. `@PreDestroy` Examples

Use `@PreDestroy` for **your own application beans** to release resources before application shutdown.

## A. Custom JDBC Connection


@Service
public class JdbcConnectionService {

    private Connection connection;

    @PostConstruct
    public void init() throws Exception {
        connection = DriverManager.getConnection(
                "jdbc:mysql://localhost:3306/employees",
                "root",
                "password");

        System.out.println("DB Connection Created");
    }

    @PreDestroy
    public void cleanup() throws Exception {
        if (connection != null) {
            connection.close();
            System.out.println("DB Connection Closed");
        }
    }
}

Suppose a payment SDK:


public class PaymentSDK {

    public void initialize() {
        System.out.println("Payment SDK initialized");
    }

    public void disconnect() {
        System.out.println("Payment SDK disconnected");
    }
}


Spring Configuration:


@Configuration
public class AppConfig {

    @Bean(initMethod = "initialize",
          destroyMethod = "disconnect")
    public PaymentSDK paymentSDK() {

        return new PaymentSDK();
    }
}


### Real-Time Use Case

Razorpay SDK, Stripe SDK, SAP SDK.

# Interview Answer

### Use `@PreDestroy`

For application-level cleanup:

* Close JDBC connections
* Redis connections
* Mongo clients
* Executor services

### Use `DisposableBean.destroy()`

For Spring framework or infrastructure components:

* Cache managers
* Thread pools
* Event processors

### Use `destroyMethod`

For third-party or legacy classes:

* Payment SDKs
* Redis clients
* FTP clients
* JMS clients

**Execution order during shutdown:**


@PreDestroy
      ↓
DisposableBean.destroy()
      ↓
destroyMethod()

## Why use `destroyMethod` for SDKs?

Suppose a Payment SDK provides methods:


public class PaymentSDK {

    public void initialize() {
        System.out.println("Payment SDK initialized");
    }

    public void shutdown() {
        System.out.println("Payment SDK shutdown");
    }
}


Since this class belongs to a third party, you cannot add `@PreDestroy` to it.

So configure it externally:


@Configuration
public class PaymentConfig {

    @Bean(initMethod = "initialize",
          destroyMethod = "shutdown")
    public PaymentSDK paymentSDK() {
        return new PaymentSDK();
    }
}


### Application Startup


Constructor
↓
initialize()


### Application Shutdown


shutdown()

## Real Example: AWS S3 Client


@Configuration

    public S3Client s3Client() {

                .region(Region.US_EAST_1)
                .build();

Spring invokes:


Application Start
↓
S3Client created

Application Shutdown
↓
s3Client.close()

## Why not `@PreDestroy`?

`@PreDestroy` works only if **you own the class**:


@Service
public class PaymentService {

    @PreDestroy
    public void cleanup() {
        // your cleanup logic
    }
}


But for third-party SDK classes:


public class StripeClient {
    public void close() { }
}


you cannot modify the source code. Therefore, use:


@Bean(destroyMethod = "close")

### Interview Answer

> SDK means Software Development Kit, such as Payment SDKs (Razorpay, Stripe), AWS SDK, Twilio SDK, etc. 
Since these are third-party classes and we cannot annotate them with `@PreDestroy`, 

Spring allows us to specify cleanup methods externally using `@Bean(destroyMethod = "...")`. 
This ensures that resources like HTTP connections, thread pools, 
or sockets held by the SDK are properly released during application shutdown.

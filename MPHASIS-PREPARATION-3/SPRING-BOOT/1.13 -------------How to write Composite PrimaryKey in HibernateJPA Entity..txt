In JPA/Hibernate, you can define a **Composite Primary Key** in two standard ways:

1. **`@EmbeddedId`** (recommended)
2. **`@IdClass`**

***

## Option 1: Using `@EmbeddedId` (Preferred)

### Step 1: Create a Composite Key Class

```java
import java.io.Serializable;
import jakarta.persistence.Embeddable;

@Embeddable
public class EmployeeProjectId implements Serializable {

    private Long employeeId;

    private Long projectId;

    public EmployeeProjectId() {
    }

    public EmployeeProjectId(Long employeeId, Long projectId) {
        this.employeeId = employeeId;
        this.projectId = projectId;
    }

    // Getters and Setters

    // equals() and hashCode()
}
```

### Step 2: Use it in the Entity

```java
import jakarta.persistence.*;

@Entity
@Table(name = "employee_project")
public class EmployeeProject {

    @EmbeddedId
    private EmployeeProjectId id;

    @Column(name = "assigned_date")
    private LocalDate assignedDate;

    // Getters and Setters
}
```

### Usage

```java
EmployeeProjectId id =
        new EmployeeProjectId(100L, 200L);

EmployeeProject employeeProject = new EmployeeProject();
employeeProject.setId(id);

entityManager.persist(employeeProject);
```

***

## Option 2: Using `@IdClass`

### Step 1: Create Id Class

```java
import java.io.Serializable;

public class EmployeeProjectId implements Serializable {

    private Long employeeId;
    private Long projectId;

    public EmployeeProjectId() {
    }

    // equals() and hashCode()
}
```

### Step 2: Entity

```java
import jakarta.persistence.*;

@Entity
@IdClass(EmployeeProjectId.class)
@Table(name = "employee_project")
public class EmployeeProject {

    @Id
    private Long employeeId;

    @Id
    private Long projectId;

    @Column(name = "assigned_date")
    private LocalDate assignedDate;

    // Getters and Setters
}
```

***

## Composite Key with Relationships

A common scenario is a join table:

```java
@Embeddable
public class OrderItemId implements Serializable {

    private Long orderId;
    private Long productId;

    // equals() and hashCode()
}
```

```java
@Entity
public class OrderItem {

    @EmbeddedId
    private OrderItemId id;

    @ManyToOne
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer quantity;
}
```

`@MapsId` maps the foreign keys directly into the composite key.

***

## Important Requirements

For the Composite Key class:

✅ Must implement `Serializable`

✅ Must have a no-arg constructor

✅ Must override `equals()` and `hashCode()`

✅ Field names in `@IdClass` must exactly match the entity's `@Id` fields

***

## Which One Should You Use?

* Use **`@EmbeddedId`** for new projects and complex keys.
* Use **`@IdClass`** when working with legacy schemas or when you want direct access to key fields from the entity.

Most modern Spring Boot + Hibernate applications prefer **`@EmbeddedId` + `@MapsId`** because it keeps the key encapsulated and 
works well with entity relationships.

---------------------------------------------------------------------------------

### What is `@MapsId` in JPA?

`@MapsId` is used when an entity's **primary key contains a foreign key**. It tells JPA:

> "Use the ID from the associated entity as part of my primary key."

This is very common with **composite primary keys**.

***

## Example: `OrderItem` with Composite Key

Suppose:

```text
Order
  |
  | (1)
  |
OrderItem
  |
  | (Many)
  |
Product
```

The table:

```sql
ORDER_ITEM
-----------
order_id    PK, FK
product_id  PK, FK
quantity
```

Composite key:

```java
(order_id, product_id)
```

***

## Step 1: Composite Key Class

```java
@Embeddable
public class OrderItemId implements Serializable {

    private Long orderId;
    private Long productId;

    // Constructors

    // equals() and hashCode()
}
```

***

## Step 2: Entity Using `@MapsId`

```java
@Entity
@Table(name = "order_item")
public class OrderItem {

    @EmbeddedId
    private OrderItemId id;

    @ManyToOne
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer quantity;
}
```

***

## How `@MapsId("orderId")` Works

```java
@MapsId("orderId")
private Order order;
```

means:

> Map `order.getId()` into `OrderItemId.orderId`.

Similarly:

```java
@MapsId("productId")
private Product product;
```

means:

> Map `product.getId()` into `OrderItemId.productId`.

You don't manually populate the composite key values.

***

## Persist Example

```java
Order order = entityManager.find(Order.class, 1L);
Product product = entityManager.find(Product.class, 101L);

OrderItem item = new OrderItem();

item.setOrder(order);
item.setProduct(product);
item.setQuantity(5);

entityManager.persist(item);
```

Hibernate automatically creates:

```text
OrderItemId(
    orderId = 1,
    productId = 101
)
```

and inserts:

```sql
INSERT INTO order_item
(order_id, product_id, quantity)
VALUES
(1, 101, 5);
```

***

## Without `@MapsId`

You would need to manually keep IDs synchronized:

```java
OrderItemId id = new OrderItemId();
id.setOrderId(order.getId());
id.setProductId(product.getId());

item.setId(id);
item.setOrder(order);
item.setProduct(product);
```

This duplicates data and can lead to inconsistencies.

***

## Single Key Example

`@MapsId` is also used with a shared primary key.

### User

```java
@Entity
public class User {

    @Id
    private Long id;
}
```

### UserProfile

```java
@Entity
public class UserProfile {

    @Id
    private Long id;

    @OneToOne
    @MapsId
    @JoinColumn(name = "id")
    private User user;
}
```

Here:

```text
User.id = UserProfile.id
```

Both tables share the same primary key.

## Interview Answer

> `@MapsId` is used when a foreign key is part of an entity's primary key. It maps an associated entity's identifier into an `@EmbeddedId` or primary key field. In composite key scenarios, it eliminates the need to manually set foreign key values inside the embedded key and keeps the relationship and primary key synchronized. A common use case is a join entity such as `OrderItem`, where `(order_id, product_id)` forms the composite primary key.

### Product Entity

### Composite Key

```java
import jakarta.persistence.Embeddable;
import java.io.Serializable;
import java.util.Objects;

@Embeddable
public class OrderItemId implements Serializable {

    private Long orderId;
    private Long productId;

    public OrderItemId() {
    }

    public OrderItemId(Long orderId, Long productId) {
        this.orderId = orderId;
        this.productId = productId;
    }

    // getters/setters

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof OrderItemId)) return false;

        OrderItemId that = (OrderItemId) o;

        return Objects.equals(orderId, that.orderId)
                && Objects.equals(productId, that.productId);
    }

    @Override
    public int hashCode() {
        return Objects.hash(orderId, productId);
    }
}
```

***

### OrderItem Entity

```java
@Entity
@Table(name = "order_item")
public class OrderItem {

    @EmbeddedId
    private OrderItemId id = new OrderItemId();

    @ManyToOne(fetch = FetchType.LAZY)
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer quantity;

    // getters/setters
}
```

***

### Save Data

```java

Product product = entityManager.find(Product.class, 100L);

OrderItem item = new OrderItem();

item.setOrder(order);
item.setProduct(product);
item.setQuantity(10);

entityManager.persist(item);
```

Hibernate automatically populates:

```java

id.productId = product.getId();
```

Generated SQL:

```sql
insert into order_item
(order_id, product_id, quantity)
values
(1, 100, 10);
```

# 2. How does `@MapsId` differ from `@JoinColumn`?

This is a very common interview question.

## `@JoinColumn`

Defines the foreign key column.

```java
@ManyToOne
@JoinColumn(name = "order_id")
private Order order;
```

This only tells Hibernate:

> Use `order_id` as the foreign key column.

It does **not** say anything about primary keys.

***

## `@MapsId`

```java
@ManyToOne
@MapsId("orderId")
@JoinColumn(name = "order_id")
private Order order;
```

This tells Hibernate:

> The foreign key `order_id` is also mapped to the `orderId` field inside my primary key.

***

### Without `@MapsId`

You must manually populate the ID.

```java
OrderItemId id = new OrderItemId();

id.setOrderId(order.getId());
id.setProductId(product.getId());

item.setId(id);
item.setOrder(order);
item.setProduct(product);
```

***

### With `@MapsId`

Only:

```java
item.setOrder(order);
item.setProduct(product);
```

Hibernate builds the composite key automatically.

***

## Interview Answer

### `@JoinColumn`

* Specifies which database column acts as FK.
* Defines relationship mapping.

### `@MapsId`

* Connects a relationship to a primary key field.
* Used when FK is part of PK.
* Eliminates duplication of ID values.

Correct. `@EmbeddedId` and `@Embeddable` themselves do **not require foreign keys**.

I think the confusion is between:

1. **Composite Primary Key** (`@Embeddable` + `@EmbeddedId`)
2. **Composite Primary Key that includes Foreign Keys** (`@MapsId`)

***

## Case 1: Composite Key WITHOUT Foreign Keys

You can use `@EmbeddedId` completely independently of relationships.

```java
@Embeddable
public class EmployeeId implements Serializable {

    private String companyCode;
    private Long employeeNumber;

    // equals & hashCode
}
```

```java
@Entity
public class Employee {

    @EmbeddedId
    private EmployeeId id;

    private String name;
}
```

Here:

```text
PK = (companyCode, employeeNumber)
```

No foreign key exists.

No `@MapsId` needed.

***

## Case 2: Composite Key WITH Foreign Keys

Suppose:

```text
ORDER_ITEM

order_id   PK, FK
product_id PK, FK
quantity
```

Now the primary key itself contains foreign keys.

```java
@Embeddable
public class OrderItemId {

    private Long orderId;
    private Long productId;
}
```

```java
@Entity
public class OrderItem {

    @EmbeddedId
    private OrderItemId id;

    @ManyToOne
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;
}
```

Here:

```text
PK = (order_id, product_id)

and

order_id  -> FK to Order
product_id -> FK to Product
```

Since the PK fields are also FKs, `@MapsId` is useful.

***

## Why `@MapsId`?

Without `@MapsId`:

```java
item.getId().setOrderId(order.getId());
item.getId().setProductId(product.getId());

item.setOrder(order);
item.setProduct(product);
```

You're storing the same information twice:

```text
id.orderId
order.id

id.productId
product.id
```

`@MapsId` keeps them synchronized automatically.

***

## The Relationship

```text
@Embeddable
       ↓
Defines composite key structure

@EmbeddedId
       ↓
Uses that composite key in Entity

@MapsId
       ↓
Optional
Used only when some field in the composite key comes from a relationship
```

### Interview Answer

> `@Embeddable` and `@EmbeddedId` are used to define and use a composite primary key. They do not imply any foreign-key relationship. `@MapsId` is an additional annotation used when one or more fields of that composite key are derived from associated entities (foreign keys). So `@MapsId` is not required for every `@EmbeddedId`; it is only needed when the primary key contains foreign-key columns.

--------------------

Exactly. `@MapsId` is **not tied to `@EmbeddedId` only**. The key idea is:

> `@MapsId` is useful when a PK field and an FK field are the same column, regardless of whether you use `@EmbeddedId` or `@IdClass`.

With `@IdClass`, the primary key fields are declared directly in the entity, so often you don't even need `@MapsId`.

## `@IdClass` Example (No `@MapsId`)

```java
@Entity
@IdClass(OrderItemId.class)
public class OrderItem {

    @Id
    @Column(name = "order_id")
    private Long orderId;

    @Id
    @Column(name = "product_id")
    private Long productId;

    @ManyToOne
    @JoinColumn(name = "order_id",
            insertable = false,
            updatable = false)
    private Order order;

    @ManyToOne
    @JoinColumn(name = "product_id",
            insertable = false,
            updatable = false)
    private Product product;
}
```

Here:

```text
order_id   PK + FK
product_id PK + FK
```

The PK values are already explicit fields:

```java
private Long orderId;
private Long productId;
```

So JPA already knows what they are.

No `@MapsId` is necessary.

***

## Why is `@MapsId` More Common with `@EmbeddedId`?

With `@EmbeddedId`, the PK fields are hidden inside another object:

```java
@EmbeddedId
private OrderItemId id;
```

JPA needs to know:

```text
Order.id  -> id.orderId
Product.id -> id.productId
```

That's exactly what `@MapsId` provides.

```java
@ManyToOne
@MapsId("orderId")
private Order order;
```

***

## Can `@MapsId` Be Used with `@IdClass`?

Yes.

Example:

```java
@Entity
@IdClass(OrderItemId.class)
public class OrderItem {

    @Id
    private Long orderId;

    @Id
    private Long productId;

    @ManyToOne
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;
}
```

But this is less common because:

* The PK fields already exist as entity fields.
* `@MapsId` adds little value.
* Most developers simply use `@JoinColumn` with `insertable=false, updatable=false`.

***

## Interview Answer

If asked:

> Why is `@MapsId` mostly seen with `@EmbeddedId` and not with `@IdClass`?

Answer:

> With `@EmbeddedId`, the primary key fields are encapsulated inside an embeddable object, so `@MapsId` is used to map relationship IDs into specific fields of that embedded key. With `@IdClass`, the primary key fields are already present directly in the entity, so JPA can map them directly and `@MapsId` is usually unnecessary, though it is still supported.


-------------

In practice, **rarely**.

`@MapsId` shines with `@EmbeddedId` because it maps a relationship's ID into a field inside the embedded key. With `@IdClass`, the ID fields already exist directly on the entity, so there's usually nothing extra to map.

## Typical `@IdClass` (No `@MapsId`)


## When `@MapsId` with `@IdClass` Makes Sense

Use it when you want the relationship itself to populate the primary-key fields automatically.

Instead of:

```java
item.setOrderId(order.getId());
item.setProductId(product.getId());
```

you want:

```java
item.setOrder(order);
item.setProduct(product);
```

and have JPA derive the PK values.

Example:

```java
@IdClass(OrderItemId.class)
@Entity
public class OrderItem {

    @Id
    private Long orderId;

    @Id
    private Long productId;

    @ManyToOne
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;
}
```

Here:

```text
Order.id    -> orderId
Product.id  -> productId
```

are synchronized automatically.

## Interview Answer

> With `@IdClass`, `@MapsId` is usually unnecessary because the primary-key fields are already declared directly in the entity. You use `@MapsId` when you want a relationship's identifier to automatically populate one or more primary-key fields, especially in shared primary-key associations or when the composite key contains foreign-key columns and you want to avoid manually copying IDs from the parent entities. For most composite-key mappings, `@EmbeddedId + @MapsId` is the cleaner and more commonly used approach.


------------------------------


Yes, that's the key point.

Both **`@EmbeddedId`** and **`@IdClass`** are only mechanisms for defining a **composite primary key**. Neither of them inherently implies foreign keys.

### `@EmbeddedId`

```java
@Embeddable
class EmployeeId {
    private String companyCode;
    private Long employeeNo;
}

@Entity
class Employee {

    @EmbeddedId
    private EmployeeId id;
}
```

Here:

```text
(companyCode, employeeNo)
```

is just a composite primary key. No foreign key involved.

***

### `@IdClass`

```java
@IdClass(EmployeeId.class)
@Entity

    @Id
    private String companyCode;

    @Id
    private Long employeeNo;
}
```

Same database result:

```text
(companyCode, employeeNo)
```

Again, no foreign key involved.

## Where does `@MapsId` come in?

`@MapsId` is unrelated to how you define the composite key (`@EmbeddedId` vs `@IdClass`).

It becomes useful only when **a primary-key column is also a foreign-key column**.

Example:

```text
ORDER_ITEM

order_id    PK + FK
product_id  PK + FK
```

Now the primary key columns happen to reference other tables.

```java
@ManyToOne
@MapsId("orderId")
private Order order;
```

means:

> Take `order.getId()` and use it as the value of the PK field `orderId`.

***

## Think of them as separate concepts

### Concept 1: How do I define a composite PK?

Choose one:

```java
@EmbeddedId
```

or

```java
@IdClass
```

***

### Concept 2: Does my PK contain FK columns?

If **No**:

```java
Employee(companyCode, employeeNo)
```

No `@MapsId`.

If **Yes**:

```java
OrderItem(order_id, product_id)
```

where both columns are foreign keys.

Then `@MapsId` may be useful.

***

### Interview Answer

> `@EmbeddedId` and `@IdClass` are only strategies for mapping composite primary keys. Neither requires foreign keys. `@MapsId` is a separate annotation used when a primary-key field is derived from an entity relationship (foreign key). So the need for `@MapsId` depends on the database model, not on whether you use `@EmbeddedId` or `@IdClass`.

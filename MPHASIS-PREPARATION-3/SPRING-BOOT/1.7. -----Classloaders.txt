		1. ClassNotFoundException: JVM will not find the class at compile time. Due to Class is not available in the classpath.
		
		2. NoClassDefFoundError: Occurs when a class was available during compilation but is missing at runtime.
		
								 **Reason:** The required class or dependency is not present during execution. 
								 
		3. VerifyError: Occurs when JVM bytecode verification fails.**Reason:** Invalid or corrupted bytecode.
		
		4. UnsupportedClassVersionError: Occurs when a class is compiled with a newer Java version than the JVM running it.
		
		5. ExceptionInInitializerError: Occurs when an exception is thrown during static initialization.  10/0 in static block.
		
Custom Class Loader: extends ClassLoader class. Ex:

Memory Leaks Related to Class Loaders:

		ClassLoader leaks are common in enterprise applications and application servers, especially during redeployments.
		
		1. Static Variable Leaks:Static objects retain references and prevent class unloading.
		
		2. ThreadLocal Leaks: Objects stored in `ThreadLocal` remain attached to threads if not removed.threadLocal.remove();
		
		3. Running Threads: Background threads started by an application may continue running after deployment. executorService.shutdown();
		
		4. JDBC Driver Leaks: JDBC drivers registered with `DriverManager` can keep references to application class loaders.
		
							  Deregister JDBC drivers. DriverManager.deregisterDriver(driver);
		
		5. Cached Objects: Objects stored in caches may retain references to loaded classes.
		
		6. Stop background threads: executorService.shutdown();
		
		7. Close connection pools: Always close the connection pool during application shutdown:
		
									hikariDataSource.close();
									
									This releases:

										* Database connections
										* Worker threads
										* Internal caches
										* ClassLoader references
		
### What is the biggest performance issue related to Custom Class Loaders?

		Creating too many custom class loaders can:

			* Increase memory consumption
			* Cause Metaspace growth
			* Prevent class unloading
			* Lead to ClassLoader leaks
		
		This is a common problem in enterprise applications, web servers, and application servers that frequently redeploy applications.

### Improve Class Loading Performance

		✅ Load classes lazily whenever possible

		✅ Reuse existing class loaders

		✅ Avoid creating excessive custom class loaders

		✅ Reduce unnecessary dynamic class loading

		✅ Avoid excessive reflection

		✅ Use Class Data Sharing (CDS)

		✅ Minimize heavy static initialization blocks

		✅ Monitor Metaspace usage

		✅ Prevent ClassLoader leaks
		
		✅ Avoid long-lived static references. 
		
		✅ Use dependency injection instead of global static state
		
		✅ Heap Dump Analysis: Heap dumps help identify memory leaks and retained ClassLoaders.

			### Popular Tools

				* Eclipse Memory Analyzer (MAT)
				* VisualVM
				* JProfiler
				* YourKit
		
		✅ Use `WeakReference` when suitable

# 2. How Does CDS (Class Data Sharing) Improve Performance?

### What is CDS?

**Class Data Sharing (CDS)** is a JVM feature that stores preprocessed class metadata in a shared archive file.

	Instead of every JVM:

		1. Loading classes
		2. Verifying bytecode
		3. Linking classes
		4. Creating metadata

from scratch, it can reuse the shared archive.

***

### Without CDS

Every JVM startup performs:


Load Class
   ↓
Verify
   ↓
Prepare
   ↓
Resolve
   ↓
Initialize


for thousands of classes.

This increases startup time and memory usage.

***

### With CDS


Shared Archive
      ↓
Mapped directly into memory
      ↓
Faster startup
      ↓
Less memory usage


Multiple JVMs can share class metadata.

***

### Benefits

#### Faster Startup

Classes are already processed and archived.


Application starts faster


***

#### Lower Memory Consumption

Common JDK classes can be shared across JVM processes.


JVM A
JVM B
JVM C
   ↓
Shared Metadata


Less duplicated memory.

***

#### Better Container Performance

CDS is extremely useful for:

* Spring Boot Applications
* Microservices
* Docker Containers
* Kubernetes Deployments

where fast startup matters.

***

### Interview Answer

**CDS improves JVM startup performance by storing preprocessed class metadata in a shared archive and reusing it across JVM instances. This reduces class loading overhead, lowers memory consumption, and speeds application startup.**

*****Multiple JVMs** → Faster startup 

How to Enable CDS?

### JDK 12+

CDS is generally enabled by default because a default archive is shipped with the JDK.


java -Xshare:off MyApplication

If CDS is active, you'll see messages similar to:

Opened archive ...
Using shared archive ...

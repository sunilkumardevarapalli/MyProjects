To verify **SQL query performance**, interviewers usually expect you to discuss **execution plans, indexing, and runtime statistics**.

### 1. Check Execution Plan (Most Important)

The execution plan shows how the database executes your query.

**SQL Server**

```sql
SET SHOWPLAN_ALL ON;
GO

SELECT * FROM Employee WHERE emp_id = 100;

GO
SET SHOWPLAN_ALL OFF;
```

Or in SSMS:

* Press **Ctrl + M** (Include Actual Execution Plan)
* Run the query
* Review:
  * Table Scan
  * Index Scan
  * Index Seek
  * Estimated Cost

✅ Prefer **Index Seek** over **Table Scan**.

***

### 2. Check Query Execution Time

**SQL Server**

```sql
SET STATISTICS TIME ON;

SELECT * FROM Employee WHERE emp_id = 100;

SET STATISTICS TIME OFF;
```

Output:

```text
SQL Server Execution Times:
CPU time = 0 ms,
elapsed time = 2 ms.
```

***

### 3. Check I/O Statistics

Measures logical and physical reads.

```sql
SET STATISTICS IO ON;

SELECT * FROM Employee WHERE emp_id = 100;

SET STATISTICS IO OFF;
```

Example:

```text
Table 'Employee'.
logical reads 3
physical reads 0
```

✅ Lower reads = Better performance.

***

### 4. Verify Index Usage

Check whether indexes are being used.

```sql
CREATE INDEX idx_emp_id
ON Employee(emp_id);
```

Then review the execution plan.

**Bad**

```text
Table Scan
```

**Good**

```text
Index Seek
```

***

### 5. Look for Common Performance Issues

#### Avoid SELECT \*

```sql
SELECT emp_name
FROM Employee;
```

instead of

```sql
SELECT *
FROM Employee;
```

#### Avoid Functions on Indexed Columns

Bad:

```sql
SELECT *
FROM Employee
WHERE YEAR(join_date) = 2025;
```

Good:

```sql
SELECT *
FROM Employee
WHERE join_date >= '2025-01-01'
AND join_date < '2026-01-01';
```

#### Avoid Leading Wildcards

Bad:

```sql
WHERE emp_name LIKE '%Sunil'
```

Good:

```sql
WHERE emp_name LIKE 'Sunil%'
```

***

### 6. Compare Before and After Optimization

Example:

```sql
SELECT *
FROM Orders
WHERE customer_id = 100;
```

Before Index:

```text
Execution Time: 800 ms
Logical Reads: 10000
Table Scan
```

After Index:

```sql
CREATE INDEX idx_customer
ON Orders(customer_id);
```

Results:

```text
Execution Time: 5 ms
Logical Reads: 20
Index Seek
```

***

### Interview Answer (Short)

> To verify SQL query performance, I analyze the execution plan, check whether Index Seek or Table Scan is used, measure execution time using `SET STATISTICS TIME`, analyze I/O using `SET STATISTICS IO`, and optimize the query with proper indexing, avoiding unnecessary scans, `SELECT *`, and functions on indexed columns. This helps reduce CPU usage, logical reads, and overall execution time.

********************************************************************************************************************

	Defination:
	-----------

		CompletableFuture is a feature introduced in Java 8 that allows us to write 

		Asynchronous, 
		Chaining,
		Combine multiple tasks,
		
		Blocking, 
		Callbacks, 
		Exeption Handling without blocking the main thread,
		
		CompletableFuture internal working (ForkJoinPool)
		
A common interview question is: "When to use thenCompose vs thenCombine?"

Use thenCompose when the second call depends on the first call's result.
Use thenCombine when both calls are independent and can run in parallel.


thenCombine():  Used when two independent futures run in parallel and you want to combine their results.

thenCompose(): Used when the second async call depends on the first result.

				Suppose payment can only be fetched after getting the order.
				
exceptionally(): Used for fallback recovery.

handle(): Used when you want both:

			Success result
			Exception result
			
whenComplete():Used for auditing/logging.

				It does not change the result.
				
thenApply()		Transform result (A -> B)

thenCompose()	Chain dependent async calls

thenCombine()	Merge independent futures

allOf()			Wait for all futures

exceptionally()	Fallback on error

handle()		Process success or error

whenComplete()	Logging/Auditing after completion

	Use thenCombine when both calls are independent and can run in parallel.

	Why do we need it? Future
	-------------------------
	
		Before Java 8, we used Future.

			Problem with Future:

				Cannot chain tasks
				No easy exception handling
				Blocking using .get()

			CompletableFuture solves all these problems.
			
			Important Methods of Future
			Method			Purpose
			
			get()			Wait and get result (blocking)
			cancel()		Cancel task
			isDone()		Check if completed
			isCancelled()	Check if cancelled
						
			Major Problems with Future 

				❌ get() is blocking

				❌ Cannot chain tasks

				❌ No easy exception handling

				❌ Cannot combine multiple futures easily

				❌ No callback support
			
	Note: If we call get() or join() on CompletableFuture, the current thread becomes blocking until the result is available.

		get: checked exception
		join: unchecked exception
	
When NOT to Use CompletableFuture?

	When using full reactive frameworks like Spring WebFlux

	When using Project Reactor (Mono, Flux)

	For simple synchronous logic

Why provide a custom Executor?

For production systems, it's usually preferred because:

You control thread count.
You can isolate workloads.
You avoid exhausting the common pool.
You can configure queue size and rejection policy.

Example:

ThreadPoolExecutor executor =
    new ThreadPoolExecutor(
        50,       // core threads
        100,      // max threads
        60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(1000)
    );

Design Review Answer

"By default, CompletableFuture uses ForkJoinPool.commonPool(). However, when an ExecutorService is explicitly supplied, all async tasks are executed by that ExecutorService instead. In our implementation, we use a dedicated ExecutorService to control concurrency and prevent overloading the shared ForkJoinPool."

That's the answer most architects/reviewers expect.

If you're explaining a Completable Feature (for example, a feature that returns a CompletableFuture or supports asynchronous completion), the key point is:

It is usually an internal async implementation detail, but its purpose may be external as well.

You can explain it like this:

"The feature is implemented asynchronously using a completable pattern, which allows the caller to continue execution without blocking. Internally, it improves scalability and responsiveness by processing work in the background thread/executor. Externally, it provides a non-blocking API and allows consumers to react when the operation completes."

When to focus on Internal Async

Discuss:

Non-blocking execution
Better resource utilization
Parallel processing
Reduced response waiting time
Thread/executor management

Example:

"We use CompletableFuture internally so long-running operations do not block the request thread."

When to focus on External Purpose

Discuss:

What benefit users/client applications get
Faster UI responsiveness
Callbacks/chaining
Integration with other async workflows

Example:

"Consumers can continue their workflow and handle the result once the operation completes instead of waiting synchronously."

Simple Interview/Design Review Answer

"It serves both purposes. Internally, it enables asynchronous processing and efficient resource usage. Externally, it exposes a non-blocking mechanism that improves responsiveness and allows callers to compose dependent operations."

This is usually the safest and most complete explanation in design reviews or architecture discussions.

Key Difference
thenCompose

Used for dependent async operations.

Order
   ↓
Payment
   ↓
Shipment


One waits for another.

Example:

Get Order
   ↓
Order contains customerId
   ↓
Get Customer
   ↓
Customer contains accountId
   ↓
Get Payment

thenCombine

Used for independent async operations.

Order
Payment
Shipment


All run at the same time and later merge results.

Interview rule:

thenCompose() = Async chaining (A → B → C)
thenCombine() = Async parallel execution (A + B → Result)

Because these calls are independent:

orderRepo.findById(id)

paymentService.getPayment(id)

Why thenCombine() Here?

Because these calls are independent:

orderRepo.findById(id)

paymentService.getPayment(id)

shipmentService.getShipment(id)


None depends on another's result, so they can run in parallel and later be merged using thenCombine().

If shipment needed a value from order, you'd use thenCompose() instead of thenCombine().


----------------------------------------------------


Using the same Order → Payment → Shipment scenario, here's how each CompletableFuture method is typically used.

1. thenCombine()

Used when two independent futures run in parallel and you want to combine their results.

Example
public CompletableFuture<OrderPaymentResponse> getOrderAndPayment(Long id) {

    CompletableFuture<Order> orderFuture =
            CompletableFuture.supplyAsync(() -> orderRepo.findById(id));

    CompletableFuture<Payment> paymentFuture =
            CompletableFuture.supplyAsync(() -> paymentService.getPayment(id));

    return orderFuture.thenCombine(
            paymentFuture,
            (order, payment) ->
                    new OrderPaymentResponse(order, payment)
    );
}

Flow
               Request
                   |
        ---------------------
        |                   |
   Order Future      Payment Future
        |                   |
        -------thenCombine---
                   |
                   V
       OrderPaymentResponse

Response DTO
@AllArgsConstructor
@Data
public class OrderPaymentResponse {
    private Order order;
    private Payment payment;
}

2. thenCompose()

Used when the second async call depends on the first result.

Suppose payment can only be fetched after getting the order.

Without thenCompose
CompletableFuture<CompletableFuture<Payment>>


You get nested futures.

With thenCompose
public CompletableFuture<Payment> getPaymentFromOrder(Long orderId) {

    return CompletableFuture
            .supplyAsync(() -> orderRepo.findById(orderId))
            .thenCompose(order ->
                    CompletableFuture.supplyAsync(
                            () -> paymentService.getPayment(order.getOrderId())
                    ));
}

Flow
Fetch Order
    |
    V
Order Received
    |
thenCompose
    |
    V
Fetch Payment
    |
    V
Payment Result

Real Use Case
orderRepo.findById(id)
          -> getCustomerId()
          -> call customer service
          -> call payment service


Each step depends on the previous result.

3. exceptionally()

Used for fallback recovery.

Suppose payment service fails.

public CompletableFuture<Payment> getPayment(Long orderId) {

    return CompletableFuture
            .supplyAsync(() -> paymentService.getPayment(orderId))
            .exceptionally(ex -> {

                System.out.println("Exception: "
                        + ex.getMessage());

                return new Payment(
                        0L,
                        "FAILED"
                );
            });
}

Flow
Call Payment Service
        |
       FAIL
        |
exceptionally()
        |
        V
Default Payment Response

Output

Instead of:

CompletionException


You get:

Payment(
   paymentId=0,
   status="FAILED"
)

4. handle()

Used when you want both:

Success result
Exception result

in one place.

public CompletableFuture<Payment> getPayment(Long orderId) {

    return CompletableFuture
            .supplyAsync(() -> paymentService.getPayment(orderId))
            .handle((result, ex) -> {

                if (ex != null) {

                    System.out.println("Error = "
                            + ex.getMessage());

                    return new Payment(
                            0L,
                            "ERROR"
                    );
                }

                return result;
            });
}

Difference
exceptionally

Has only exception.

.exceptionally(ex -> fallback)

handle

Has both result and exception.

.handle((result, ex) -> ...)

Success Scenario
result = Payment(SUCCESS)
ex = null


Returns:

result

Failure Scenario
result = null
ex = RuntimeException


Returns:

new Payment(0L,"ERROR")

5. whenComplete()

Used for auditing/logging.

It does not change the result.

public CompletableFuture<Payment> getPayment(Long orderId) {

    return CompletableFuture
            .supplyAsync(() -> paymentService.getPayment(orderId))
            .whenComplete((result, ex) -> {

                if (ex == null) {
                    System.out.println(
                            "Payment Success : "
                                    + result.getStatus());
                }
                else {
                    System.out.println(
                            "Payment Failed : "
                                    + ex.getMessage());
                }
            });
}

Flow
Get Payment
      |
      V
whenComplete
      |
      V
Logging Only
      |
      V
Original Result Returned


Unlike handle(), it cannot replace the result.

Complete Enterprise Example

Imagine:

1. Fetch Order
2. Fetch Payment
3. Fetch Shipment
4. Combine all
5. Handle errors
6. Log success/failure

public CompletableFuture<OrderResponse> getOrderDetails(Long id) {

    CompletableFuture<Order> orderFuture =
            CompletableFuture.supplyAsync(
                    () -> orderRepo.findById(id));

    CompletableFuture<Payment> paymentFuture =
            CompletableFuture.supplyAsync(
                    () -> paymentService.getPayment(id))
                    .exceptionally(ex ->
                            new Payment(0L, "FAILED"));

    CompletableFuture<Shipment> shipmentFuture =
            CompletableFuture.supplyAsync(
                    () -> shipmentService.getShipment(id));

    return orderFuture
            .thenCombine(paymentFuture,
                    (order, payment) ->
                            new OrderPaymentResponse(order, payment))

            .thenCombine(shipmentFuture,
                    (op, shipment) ->
                            new OrderResponse(
                                    op.getOrder(),
                                    op.getPayment(),
                                    shipment))

            .whenComplete((result, ex) -> {

                if (ex == null) {
                    System.out.println(
                            "Order processed successfully");
                } else {
                    System.out.println(
                            "Order processing failed");
                }
            });
}

Interview Shortcut
Method	PurposethenApply()	Transform result (A -> B)
thenCompose()	Chain dependent async calls
thenCombine()	Merge independent futures
allOf()	Wait for all futures
exceptionally()	Fallback on error
handle()	Process success or error
whenComplete()	Logging/Auditing after completion

A common interview question is: "When to use thenCompose vs thenCombine?"

Use thenCompose when the second call depends on the first call's result.
Use thenCombine when both calls are independent and can run in parallel.

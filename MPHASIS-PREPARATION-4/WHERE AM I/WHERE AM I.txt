Kubernates:

	API Server:
		
		The API Server performs:

			Authentication
			Authorization
			Admission Controllers
			YAML validation
			API version validation
			Schema validation

		It also adds metadata like:

			uid
			creationTimestamp
			resourceVersion
			namespace (default if not given)
						
			metadata:
			  name: nginx-pod
			  namespace: default
			  
		API Server stores Pod object into ETCD. Pod is NOT scheduled yet. Status is Pending
		
		spec:
		  nodeName: null
---------------------------------------------------------------------------------------------------------------
		  
	Scheduler Watches API Server:  Scheduler never directly reads etcd.
	
		For Pods:  spec.nodeName == empty   Unscheduled Pods
		
		Scheduler asks API Server for:

			Available Nodes
			CPU/Memory usage
			Taints/Tolerations
			Node Affinity
			Pod Affinity/Anti-affinity
			Resource requests/limits
			Topology spread constraints
			Policies
			
		Scheduler performs two major phases:
		
			A. Filtering Phase (Predicates)
			
			B. Scoring Phase (Priorities)
			
		A. Filtering Phase (Predicates)
			
			Removes unsuitable nodes.

				Example checks:

				Enough CPU?
				Enough Memory?
				Taints matched?
				NodeSelector matched?
				Port conflicts?
				PVC available?
				Affinity rules satisfied?
			
			Suppose cluster has:

				Worker-1
				Worker-2
				Worker-3

				After filtering:

				Worker-1 ❌
				Worker-2 ✅
				Worker-3 ✅
				
		B. Scoring Phase (Priorities)

			Scheduler scores remaining nodes.

			Example scoring factors:

				Least resource usage
				Balanced resource allocation
				Image locality
				Affinity preference
				Topology balancing

			Example:

				Worker-2 = 75
				Worker-3 = 92
				
			Best node selected:
			
				Worker-3
							
	Scheduler Sends Binding to API Server:

		Scheduler does NOT directly write into node.
		
		Instead it sends:
		
			Binding Request → API Server
					
			spec:
			  nodeName: worker-3
			
	API Server Updates ETCD
	
		API Server updates Pod object in etcd.
		
		Now:
		
			spec:
              nodeName: worker-3
			  
			  Pod becomes: Scheduled
			  
----------------------------------------------------------------------------------------------------------------------------------
			  
	Kubelet Watches API Server:
	
		On Worker-3:  
		
			Kubelet watches API Server
			
		for Pods assigned to its node.	
		
		Kubelet sees:Pod assigned to worker-3
		
	Kubelet Creates Containers	
	
		Kubelet talks to Container Runtime:

			Docker
			containerd
			CRI-O

		Container runtime:

			Pulls image
			Creates container
			Starts Pod
--------------------------------------------------kube-proxy--------------------------------------------------------------		

	Pod Networking Setup
	
		kube-proxy + CNI plugin:

			Assign Pod IP
			Configure networking
			Configure Service routing
 
	Kubelet Reports Status:

		Kubelet sends Pod status to API Server:

			ContainerCreating
			Running
			Failed
			Succeeded

		API Server stores latest status in etcd.

	Real-Time Analogy

		Think of:

			API Server = Receptionist
			ETCD = Database/File Storage
			Scheduler = Manager assigning rooms
			Kubelet = Worker executing task

		Manager (Scheduler) never directly accesses database.

		Everything goes through receptionist (API Server).

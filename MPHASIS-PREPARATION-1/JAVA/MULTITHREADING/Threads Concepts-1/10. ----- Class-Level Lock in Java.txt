## Class-Level Lock in Java

A **class-level lock** is acquired when a thread enters a `static synchronized` method.

**Important:**

* Object-level lock → lock belongs to an object
* Class-level lock → lock belongs to the Class (`.class`) object

There is only **one Class object per class loaded by JVM**.

Example:

```java
class Employee {

    static synchronized void method1() {
    }

    static synchronized void method2() {
    }
}
```

Both methods use:

```java
Employee.class
```

as the lock.

***

## Scenario 1: Two Objects Calling Static Synchronized Methods

```java
Employee e1 = new Employee();
Employee e2 = new Employee();

Thread-1 --> e1.method1()
Thread-2 --> e2.method2()
```

Many developers think:

> e1 and e2 are different objects, so both threads should execute.

Wrong ❌

Because methods are static synchronized.

Java acquires:

```java
Employee.class
```

lock.

### Execution

Thread-1 enters method1()

```text
Locks Employee.class
```

Thread-2 enters method2()

```text
Needs Employee.class lock
```

But lock is already occupied.

Result:

```text
Thread-1 --> Permission ✅
Thread-2 --> Waiting ❌
```

After Thread-1 completes:

```text
Thread-2 --> Permission ✅
```

***

## Visualization

```text
Employee.class LOCK

Thread-1 ---> method1() ✅

Thread-2 ---> method2() ❌ Waiting
```

Even though they are using different objects.

***

## Scenario 2: Different Objects + Instance Synchronized Methods

```java
class Employee {

    synchronized void m1() {
    }

    synchronized void m2() {
    }
}
```

```java
Employee e1 = new Employee();
Employee e2 = new Employee();

Thread-1 --> e1.m1()
Thread-2 --> e2.m2()
```

Locks:

```text
e1 lock
e2 lock
```

Result:

```text
Thread-1 --> Permission ✅
Thread-2 --> Permission ✅
```

No problem.

# Most Important Interview Question

## If a Static Synchronized Method Accesses Instance Variables

Example:

```java
class Employee {

    int salary = 1000;

    static synchronized void update() {
        // trying to access salary
    }
}
```

This will not compile.

Reason:

```text
Instance variable belongs to object
Static method belongs to class
```

A static method cannot directly access instance members.

Need object reference:

```java
Employee e = new Employee();
e.salary;
```

***

## If Static Synchronized Method Accesses Static Variable

Example:

```java
class Employee {

    static int count = 0;

    static synchronized void increment() {
        count++;
    }
}
```

This is perfectly valid.

Lock:

```text
Employee.class
```

Data:

```text
count
```

Both belong to class level.

No race condition.

***

# When Problems Come

The interviewers usually ask this.

***

## Case 1: Static Variable + Instance Lock

```java
class Employee {

    static int count = 0;

    synchronized void increment() {
        count++;
    }
}
```

Objects:

```java
Employee e1 = new Employee();
Employee e2 = new Employee();
```

Threads:

```java
Thread-1 --> e1.increment()
Thread-2 --> e2.increment()
```

Locks:

```text
e1 lock
e2 lock
```

Different locks.

But data:

```text
static count
```

Shared by entire class.

Both threads can modify count simultaneously.

### Problem

Race Condition ✅

Because:

```text
Different locks
Same data
```

This is a very common bug.

***

## Case 2: Instance Variable + Class Lock

Example:

```java
class Employee {

    int salary;

    static synchronized void update(Employee e) {
        e.salary++;
    }
}
```

Lock:

```text
Employee.class
```

Even if:

```java
Employee e1
Employee e2
Employee e3
```

All updates are controlled by one lock.

No race condition.

But:

### Performance Problem

Suppose:

```text
Thread-1 updating e1
Thread-2 updating e2
Thread-3 updating e3
```

These objects are unrelated.

Yet all threads wait for:

```text
Employee.class lock
```

Result:

```text
Safe ✅
Slow ❌
```

***

# When Should You Use Class-Level Lock?

Use it when protecting:

* Static variables
* Shared counters
* Application-wide caches
* Singleton objects
* Shared resources common to all objects

Example:

```java
static int totalUsers;
static int totalOrders;
static Map cache;
```

## 1-Minute Interview Answer

> A class-level lock is obtained when a thread enters a static synchronized method.
 The lock belongs to the Class object, not to any individual instance. 
 Therefore, even if two different objects call two different static synchronized methods, 
 only one thread can execute at a time because both require the same Class lock. 
 Class-level locking is mainly used to protect static data. 
 Problems arise when static variables are protected using instance locks, 
 because different objects have different locks but the static data is shared, 
 which can lead to race conditions. Conversely, using a class-level lock for instance data is safe but may 
 unnecessarily reduce concurrency and performance.

# Optimistic Lock - Complete Code Example (JPA/Hibernate)

Optimistic locking is typically implemented using the `@Version` annotation.

## Entity

```java
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Version;

@Entity
public class Account {

    @Id
    private Long id;

    private Double balance;

    @Version
    private Long version;

    public Account() {
    }

    public Account(Long id, Double balance) {
        this.id = id;
        this.balance = balance;
    }

    public Long getId() {
        return id;
    }

    public Double getBalance() {
        return balance;
    }

    public void setBalance(Double balance) {
        this.balance = balance;
    }

    public Long getVersion() {
        return version;
    }
}
```

***

# Database Table

Initially:

```text
ID    BALANCE    VERSION
------------------------
1     10000      0
```

***

# Service Example

```java
@Service
@Transactional
public class AccountService {

    @Autowired
    private AccountRepository repository;

    public void withdraw(Long accountId,double amount) {

        Account account =repository.findById(accountId).orElseThrow();

        account.setBalance(account.getBalance() - amount);

        repository.save(account);
    }
}
```

***

# Concurrent Update Scenario

## Transaction-1

Reads:

```text
ID=1
BALANCE=10000
VERSION=0
```

***

## Transaction-2

Reads:

```text
ID=1
BALANCE=10000
VERSION=0
```

***

## Transaction-1 Updates

```text
BALANCE=9000
VERSION=1
```

Success ✅

Database:

```text
ID=1
BALANCE=9000
VERSION=1
```

***

## Transaction-2 Updates

Hibernate executes:

```sql
UPDATE ACCOUNT
SET BALANCE = 8000,
    VERSION = 1
WHERE ID = 1
AND VERSION = 0;
```

But current version is:

```text
1
```

Rows updated:

```text
0
```

Hibernate throws:

```java
OptimisticLockException
```

***

# Handling OptimisticLockException

```java
try {

    repository.save(account);

} catch (ObjectOptimisticLockingFailureException ex) {

    System.out.println(
            "Record already modified by another user");

    // reload and retry
}
```

# Multithreaded Example (Conceptual)

### Thread-1

```java
Account acc =repository.findById(1L).get();

acc.setBalance(acc.getBalance() - 1000);

repository.save(acc);
```

### Thread-2

```java
Account acc =
        repository.findById(1L).get();

acc.setBalance(
        acc.getBalance() - 2000);

repository.save(acc);
```

Result:

```text
One transaction succeeds

Second transaction gets
OptimisticLockException
```

No lost updates.

# When Should You Prefer Optimistic Lock?

Prefer Optimistic Lock when:

### 1. Read Operations >> Write Operations

Example:

```text
1000 Reads
10 Updates
```

Conflicts are rare.

✅ Use Optimistic Lock

***

### 2. Employee Profile System

```text
Many users view profiles
Rarely updated
```

Example:

```text
Name
Address
Department
```

✅ Optimistic Lock

***

### 3. Insurance Policy Portal

```text
Agents mostly read policies
Occasional modifications
```

✅ Optimistic Lock

***

### 4. Product Catalog

```text
Customers view products
Admins update occasionally
```

✅ Optimistic Lock

***

### 5. CMS (Content Management)

```text
Article editing
Blog editing
```

If two editors modify the same article:

```text
Version conflict detected
```

✅ Optimistic Lock

***

### 6. Customer Master Data

```text
Customer Name
Email
Address
```

Updates are infrequent.

✅ Optimistic Lock

***

# When NOT to Use Optimistic Lock?

Avoid when conflicts are frequent.

### Flight Booking

```text
1 seat left
100 users booking
```

❌ Too many retries

Use:

```text
Pessimistic Lock
```

***

### Movie Ticket Booking

```text
Last seat booking
```

❌ Pessimistic Lock preferred

***

### Inventory Deduction

```text
Stock = 1
Many concurrent orders
```

❌ Optimistic Lock causes many conflicts and retries

***

### Banking Transactions

```text
Balance updates
Money transfer
```

❌ Usually Pessimistic Lock or database transaction locking

***

### Trading Systems

```text
High-frequency updates
```

❌ Optimistic Lock not ideal

***

# Interview Answer (1 Minute)

> Optimistic locking uses a version column, typically implemented with the `@Version` annotation in JPA/Hibernate.
 Multiple transactions can read the same record concurrently without locking it. During update, 
 Hibernate checks whether the version has changed. If another transaction has already modified 
 the record, an `OptimisticLockException` is thrown. Optimistic locking is preferred in read-heavy applications 
 where conflicts are rare, such as employee profiles, customer master data, CMS systems, insurance portals, and product catalogs. 
 It improves performance because it avoids database locking and blocking.

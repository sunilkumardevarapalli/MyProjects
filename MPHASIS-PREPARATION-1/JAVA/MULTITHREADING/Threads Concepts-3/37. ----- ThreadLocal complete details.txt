# ThreadLocal in Java – Complete Interview Guide



ThreadLocal:  `ThreadLocal` provides a **separate copy of a variable for each thread**.No sharing.

Normally, threads share variables:


	class Test {
		static int count = 0;
	}


	All threads access the same variable.

	With `ThreadLocal`:


	ThreadLocal<String> user =  new ThreadLocal<>();Now each thread has its own copy.

	Each thread gets its own value.

		Thread-1 -> Sunil
		Thread-2 -> Ravi
		Thread-3 -> Kiran
		
	No sharing.
	
	
# Why Do We Need ThreadLocal?

		Suppose multiple users send requests:

			User-1 Request
			User-2 Request
			User-3 Request

		Each request needs:

			UserId
			SessionId
			TransactionId

		If stored in a shared variable:

		String userId;

		Threads may overwrite each other's data.
		
		

# Basic Example


public class ThreadLocalDemo {

    private static ThreadLocal<String> threadLocal =
            new ThreadLocal<>();

    public static void main(String[] args) {

        Thread t1 = new Thread(() -> {

            threadLocal.set("Sunil");

            System.out.println(
                    Thread.currentThread().getName()
                    + " : "
                    + threadLocal.get());
        });

        Thread t2 = new Thread(() -> {

            threadLocal.set("Ravi");

            System.out.println(
                    Thread.currentThread().getName()
                    + " : "
                    + threadLocal.get());
        });

        t1.start();
        t2.start();
    }
}
### Output

	Thread-0 : Sunil
	Thread-1 : Ravi

	Even though both use the same variable:


threadLocal


they see different values.

# Example with Two Threads


public class Test {

    private static ThreadLocal<Integer> count =
            ThreadLocal.withInitial(() -> 0);

    public static void main(String[] args) {

        Runnable task = () -> {

            for(int i=1;i<=5;i++) {

                count.set(
                        count.get() + 1);
            }

            System.out.println(
                    Thread.currentThread().getName()
                    + " : "
                    + count.get());
        };

        new Thread(task, "T1").start();
        new Thread(task, "T2").start();
    }
}


Output:


T1 : 5
T2 : 5


Each thread gets its own count.

# ThreadLocal in Thread Pool

Very Important Interview Question.

## Problem


ExecutorService ex =
        Executors.newFixedThreadPool(1);


Thread is reused.

Task-1:


user.set("Sunil");


Task-2:


System.out.println(user.get());


May print:


Sunil


even though Task-2 never set it.

Reason:


Same thread reused


***

# Solution

Always clean ThreadLocal.


try {

    user.set("Sunil");

} finally {

    user.remove();
}


Best practice:


remove()


after completion.

***

# ThreadLocal Memory Leak

Suppose:


ThreadLocal<String> user =
        new ThreadLocal<>();


You never call:


user.remove();


In a thread pool:


Thread survives
Value survives


Memory leak possibility.

Therefore:


try {

   // work

} finally {

   user.remove();
}

# Interview Questions

### Is ThreadLocal thread-safe?

Yes.

Because each thread gets its own copy.

***

### Can one thread access another thread's ThreadLocal value?

No.

***

### Does ThreadLocal need synchronization?

No.

Each thread has independent storage.

***

### Why remove ThreadLocal values?

To prevent memory leaks, especially in thread pools.

***

# Interview Answer (1 Minute)

> ThreadLocal provides thread-specific storage, meaning each thread gets its own independent copy of a variable. 
It is commonly used for user sessions, transaction IDs, database connections, request context, and logging correlation IDs. 
Unlike synchronization, which protects shared data, ThreadLocal avoids sharing data altogether. 
Important methods are `set()`, `get()`, and `remove()`. In thread pool environments, developers should always call 
`remove()` after use to prevent memory leaks and stale data from being reused by other tasks on the same worker thread.

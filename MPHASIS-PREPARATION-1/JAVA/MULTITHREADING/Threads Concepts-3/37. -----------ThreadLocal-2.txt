# 1. ThreadLocal with `try-finally` (Best Practice)

When using `ThreadLocal`, always remove the value after use, especially in thread pools.

## Correct Example

```java
public class ThreadLocalDemo {

    private static ThreadLocal<String> userContext =new ThreadLocal<>();

    public static void main(String[] args) {

        Runnable task = () -> {

            try {

                userContext.set("Sunil");

                System.out.println(
                        Thread.currentThread().getName()
                        + " User = "
                        + userContext.get());

            } finally {

                userContext.remove();

                System.out.println(
                        Thread.currentThread().getName()
                        + " ThreadLocal Removed");
            }
        };

        new Thread(task, "T1").start();
    }
}
```

### Output

```text
T1 User = Sunil
T1 ThreadLocal Removed
```

***

# Thread Pool Example (Most Important)

```java
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ThreadLocalPoolDemo {

    private static ThreadLocal<String> user =
            new ThreadLocal<>();

    public static void main(String[] args) {

        ExecutorService ex =
                Executors.newFixedThreadPool(1);

        ex.submit(() -> {

            try {

                user.set("Sunil");

                System.out.println(
                        "Task1 : "
                        + user.get());

            } finally {

                user.remove();
            }
        });

        ex.submit(() -> {

            System.out.println(
                    "Task2 : "
                    + user.get());
        });

        ex.shutdown();
    }
}
```

### Output

```text
Task1 : Sunil
Task2 : null
```

Because:

```java
user.remove();
```

cleaned the thread-local storage.

***

# 2. How Memory Leak Happens with ThreadLocal

This is one of the most asked senior-level interview questions.

## Without remove()

```java
private static ThreadLocal<String> user =
        new ThreadLocal<>();
```

Task:

```java
user.set("Sunil");
```

But:

```java
user.remove();
```

is never called.

***

## What Happens Internally?

Every Thread has:

```text
Thread
   |
   |
ThreadLocalMap
```

```text
Thread-1

ThreadLocalMap
--------------------
user = Sunil
city = Hyderabad
```

The value remains attached to the thread.

***

## Problem in Thread Pool

```java
ExecutorService ex =
        Executors.newFixedThreadPool(1);
```

Thread:

```text
pool-1-thread-1
```

never dies.

### Task-1

```java
user.set("Sunil");
```

Stored:

```text
ThreadLocalMap

user = Sunil
```

***

### Task-2

Same worker thread reused:

```text
pool-1-thread-1
```

If cleanup wasn't done:

```java
System.out.println(user.get());
```

Output:

```text
Sunil
```

Even though Task-2 never set it.

This is called:

```text
Stale ThreadLocal Data
```

***

## Memory Leak Visualization

```text
Thread Pool

pool-1-thread-1
    |
    |
ThreadLocalMap
--------------------------------
User Object
Session Object
Connection Object
Large Cache Object
```

Worker thread stays alive.

Objects remain referenced.

GC cannot reclaim them.

Memory grows.

Possible:

```text
java.lang.OutOfMemoryError
```

***

# Memory Leak Example

```java
ThreadLocal<byte[]> cache =
        new ThreadLocal<>();

cache.set(
        new byte[50 * 1024 * 1024]);
```

50 MB.

If:

```java
cache.remove();
```

is not called

and thread pool threads keep running:

```text
50 MB remains referenced
```

Memory usage keeps increasing.

***

# Correct Solution

```java
try {

    threadLocal.set(object);

} finally {

    threadLocal.remove();
}
```

Always.

***

# 3. How to Debug ThreadLocal Issues?

## Symptom 1

Unexpected Data

Example:

```text
Request-1 User = Sunil

Request-2 User = Sunil
```

even though Request-2 belongs to Ravi.

Possible Cause:

```java
remove() missing
```

***

## Symptom 2

Memory Leak

```text
Heap usage growing continuously
```

Thread pool application.

Likely cause:

```java
ThreadLocal values retained
```

***

## Debug Technique 1: Log Thread Name

```java
System.out.println(

        Thread.currentThread().getName()
        + " -> "
        + userContext.get()
);
```

Output:

```text
pool-1-thread-1 -> Sunil
pool-1-thread-1 -> Sunil
```

Shows same thread reused.

***

## Debug Technique 2: Log Before and After

```java
try {

    user.set("Sunil");

    System.out.println(
            "Before Remove : "
            + user.get());

} finally {

    user.remove();

    System.out.println(
            "After Remove : "
            + user.get());
}
```

Output:

```text
Before Remove : Sunil
After Remove : null
```

***

## Debug Technique 3: Heap Dump Analysis

Tools:

```text
MAT (Memory Analyzer Tool)
VisualVM
JProfiler
YourKit
```

Look for:

```text
java.lang.Thread
   |
ThreadLocalMap
```

Large retained objects.

Example:

```text
Thread
 |
ThreadLocalMap
 |
CustomerCache
 |
50 MB
```

This usually indicates ThreadLocal leakage.

***

## Debug Technique 4: Thread Dump

Commands:

```bash
jstack <pid>
```

or

```bash
jcmd <pid> Thread.print
```

Useful for identifying:

```text
Long-running pool threads
```

that may still hold ThreadLocal values.

# Interview Questions

### Why use `try-finally` with ThreadLocal?

```java
try {
   threadLocal.set(value);
}
finally {
   threadLocal.remove();
}
```

To guarantee cleanup even if exceptions occur.

***

### Why is ThreadLocal dangerous in thread pools?

Because worker threads are reused. Old values may remain attached to the thread and be visible to later tasks.

***

### Can ThreadLocal cause memory leaks?

Yes.

If large objects are stored and `remove()` is not called, thread pool threads can retain references for a long time.

***

# Interview Answer

> ThreadLocal should always be used with a `try-finally` block so that `remove()` is guaranteed to execute.

 In thread pools, worker threads are reused, so failing to call `remove()` can leave stale data attached to the thread and may

 cause memory leaks. To debug ThreadLocal issues, check for unexpected data reuse across requests, log thread names and ThreadLocal

 values, analyze heap dumps with MAT or VisualVM, and inspect long-lived worker threads that may be holding references through their 
 
 ThreadLocalMap.

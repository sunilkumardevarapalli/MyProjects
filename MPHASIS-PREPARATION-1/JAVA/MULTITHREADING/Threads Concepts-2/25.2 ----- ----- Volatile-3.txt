# When Should You Avoid Using `volatile`?

A good interview answer is:

> Use `volatile` only when you need **visibility**. Avoid it when you need **atomicity**, **thread safety**, 
or **multiple operations on shared data**.

***

## 1. Avoid `volatile` for Increment/Decrement Operations

❌ Wrong

```java
volatile int count = 0;

count++;
```

Looks like one operation, but internally:

```text
Read count
Increment
Write count
```

Example:

```text
Thread-1 reads 0
Thread-2 reads 0

Thread-1 writes 1
Thread-2 writes 1
```

Expected:

```text
2
```

Actual:

```text
1
```

Race Condition occurs.

✅ Use:

```java
AtomicInteger count = new AtomicInteger();

count.incrementAndGet();
```

or

```java
synchronized
```

***

## 2. Avoid `volatile` for Bank Account Updates

❌ Wrong

```java
volatile double balance = 10000;

balance = balance - 500;
```

Multiple threads:

```text
ATM Withdrawal
GooglePay Transfer
PhonePe Payment
```

may overwrite each other's updates.

✅ Use:

```java
synchronized
```

or

```java
Lock
```

***

## 3. Avoid `volatile` for Collections

❌ Wrong

```java
volatile List<String> customers =
        new ArrayList<>();
```

```java
customers.add("Sunil");
```

Problem:

```text
ArrayList is not thread-safe
```

Volatile only makes the reference visible.

It does NOT make:

```java
add()
remove()
clear()
```

thread-safe.

✅ Use:

```java
CopyOnWriteArrayList
```

or

```java
Collections.synchronizedList()
```

***

## 4. Avoid `volatile` When Multiple Variables Must Stay Consistent

Example:

```java
volatile int x;
volatile int y;
```

Thread-1:

```java
x = 100;
y = 200;
```

Thread-2:

```java
System.out.println(x);
System.out.println(y);
```

Thread-2 may see:

```text
x = 100
y = old value
```

temporarily.

If variables must be updated together:

✅ Use synchronization.

***

## 5. Avoid `volatile` for Check-Then-Act Operations

❌ Wrong

```java
if(balance > 1000) {

    balance = balance - 1000;
}
```

Scenario:

```text
Thread-1 checks balance
Thread-2 checks balance

Both see sufficient balance
Both withdraw
```

Result:

```text
Incorrect balance
```

✅ Use synchronization.

***

## 6. Avoid `volatile` for Lazy Singleton Creation

❌ Incorrect implementation

```java
class Singleton {

    private static volatile Singleton instance;

    public static Singleton getInstance() {

        if(instance == null) {
            instance = new Singleton();
        }

        return instance;
    }
}
```

Multiple threads can still create multiple objects.

✅ Use:

```java
synchronized
```

or proper Double-Checked Locking.

***

## 7. Avoid `volatile` When Threads Must Block Each Other

Volatile does not provide:

```text
Mutual Exclusion
```

Example:

```java
volatile int seats = 1;
```

Two users booking the last seat.

Both threads may see:

```text
seats = 1
```

and both book it.

✅ Use:

```java
synchronized
```

or

```java
ReentrantLock
```

***

# When Is `volatile` a Good Choice?

✅ Shutdown Flag

```java
volatile boolean shutdown;
```

✅ Feature Toggle

```java
volatile boolean cacheEnabled;
```

✅ Configuration Refresh

```java
volatile String currentVersion;
```

✅ Application Status Flags

```java
volatile boolean serverRunning;
```

In these cases:

```text
One thread updates
Many threads read
```

No calculations are involved.

***

# Easy Rule to Remember

Use `volatile` when:

```text
Read + Write
```

Avoid `volatile` when:

```text
Read + Modify + Write
```

Examples of **Read + Modify + Write**:

```java
count++;
balance -= 500;
list.add();
map.put();
```

These need synchronization or atomic classes.

***

# Interview Answer (30 Seconds)

> Avoid using `volatile` when multiple threads perform compound operations such as increment, decrement, balance updates, 
collection modifications, or check-then-act logic. Volatile guarantees only visibility, not atomicity or mutual exclusion.
 Whenever shared data requires thread safety or consistency across multiple operations, use `synchronized`, `Lock`, 
 or atomic classes like `AtomicInteger` instead of `volatile`.

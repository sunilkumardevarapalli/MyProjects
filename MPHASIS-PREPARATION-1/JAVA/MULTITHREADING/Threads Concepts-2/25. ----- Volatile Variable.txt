# Volatile Variable in Java (Complete Interview Guide)

`volatile` is one of the most important concepts in Java Multithreading.

## What is a Volatile Variable?

A `volatile` variable tells the JVM:

> "Whenever a thread reads this variable, read it from Main Memory. Whenever a thread updates this variable, 
immediately write it to Main Memory."

Example:

```java
private volatile boolean flag = true;
```

***

# Why Do We Need Volatile?

In a multithreaded environment, each thread may keep its own local copy of variables in CPU cache.

Without `volatile`:

```text
Main Memory
    |
    | balance = 100
    |
-------------------------
|                       |
Thread-1 Cache     Thread-2 Cache
balance=100        balance=100
```

If Thread-1 changes:

```text
balance = 200
```

Thread-2 might still see:

```text
balance = 100
```

because it's reading from its cache.

This is called a **Visibility Problem**.

***

# Complete Example

```java
class SharedData {

    volatile boolean running = true;
}
```

```java
class WorkerThread extends Thread {

    private SharedData data;

    public WorkerThread(SharedData data) {
        this.data = data;
    }

    @Override
    public void run() {

        System.out.println(
                "Worker Started");

        while(data.running) {

        }

        System.out.println(
                "Worker Stopped");
    }
}
```

```java
public class Test {

    public static void main(String[] args)
            throws Exception {

        SharedData data =
                new SharedData();

        WorkerThread worker =
                new WorkerThread(data);

        worker.start();

        Thread.sleep(5000);

        System.out.println(
                "Main Thread Updating Flag");

        data.running = false;
    }
}
```

### Output

```text
Worker Started

Main Thread Updating Flag

Worker Stopped
```

# Important Interview Question

## Does Volatile Provide Thread Safety?

Answer:

```text
NO
```

This is the most important point.

***

## Example

```java
volatile int count = 0;
```

Two threads:

```java
count++;
```

Looks simple but internally:

```text
Read count
Increment
Write count
```

Three operations.

Suppose:

```text
Thread-1 reads 0

Thread-2 reads 0

Thread-1 writes 1

Thread-2 writes 1
```

Expected:

```text
2
```

Actual:

```text
1
```

Race Condition.

***

# Volatile Does NOT Prevent Race Conditions

Wrong:

```java
volatile int count;

count++;
```

Correct:

```java
synchronized
```

or

```java
AtomicInteger
```

***

# Volatile vs Synchronized

| Feature          | volatile | synchronized |
| ---------------- | -------- | ------------ |
| Visibility       | Yes      | Yes          |
| Atomicity        | No       | Yes          |
| Mutual Exclusion | No       | Yes          |
| Lock Required    | No       | Yes          |
| Performance      | Faster   | Slower       |
| Thread Safety    | Partial  | Complete     |

***

# When to Use Volatile?

Use when:

✅ Multiple threads read variable

✅ One thread updates variable

✅ No complex operations

✅ No increment/decrement

Examples:

* Shutdown flag
* Status flag
* Configuration refresh
* Feature toggle
* Ready flag

# Most Asked Interview Questions

### Is volatile thread-safe?

No.

It provides visibility but not atomicity.

***

### Does volatile prevent race conditions?

No.

Use synchronization or Atomic classes.

***

### Can volatile and synchronized be used together?

Yes.

Though it's usually unnecessary because synchronized already provides visibility and atomicity.

### When should I use volatile?

Use it for flags, status variables, and simple shared variables where only visibility is required.

# 30-Second Interview Answer

> `volatile` is a Java keyword that guarantees visibility of a variable across threads. 
When one thread updates a volatile variable, the updated value is immediately written to main memory and becomes
 visible to all other threads. However, volatile does not provide atomicity and cannot prevent race conditions.
 It is commonly used for status flags, shutdown signals, and configuration variables where only visibility is
 required and no compound operations such as increment or decrement are performed.


**Atomicity** means an operation is performed as **one indivisible unit**—either it completes entirely or not at all. No other thread can observe or interfere with it halfway through.

### Example: Not Atomic

```java
class Counter {
    int count = 0;

    void increment() {
        count++;   // Not atomic
    }
}
```

The statement:

```java
count++;
```

actually consists of multiple steps:

1. Read `count`
2. Add `1`
3. Write the new value back

If two threads execute this simultaneously, updates can be lost.

```text
Thread-1 reads 5
Thread-2 reads 5

Thread-1 writes 6
Thread-2 writes 6

Expected: 7
Actual: 6
```

### Using `volatile`

```java
volatile int count = 0;
```

`volatile` guarantees **visibility** (all threads see the latest value), but it does **not** make `count++` atomic.

So the race condition still exists.

### Using `synchronized`

```java
class Counter {
    int count = 0;

    synchronized void increment() {
        count++;
    }
}
```

Here:

* Only one thread can execute `increment()` at a time.
* The entire read-modify-write sequence becomes atomic.
* `synchronized` also provides visibility.

### Why `volatile` is Usually Unnecessary with `synchronized`

```java
private int count;

public synchronized void increment() {
    count++;
}

public synchronized int getCount() {
    return count;
}
```

Every access is synchronized, so:

* ✅ Visibility is guaranteed.
* ✅ Atomicity is guaranteed.

Adding `volatile`:

```java
private volatile int count;
```

provides no additional benefit in this case.

### Quick Summary

| Feature          | volatile | synchronized |
| ---------------- | -------- | ------------ |
| Visibility       | ✅ Yes    | ✅ Yes        |
| Atomicity        | ❌ No     | ✅ Yes        |
| Mutual Exclusion | ❌ No     | ✅ Yes        |

So when we say:

> "`synchronized` already provides visibility and atomicity"

**Atomicity means the protected block of code executes as a single, uninterrupted operation from the perspective of other threads.**

### When would you choose CountDownLatch over CyclicBarrier?

**Answer:**

  Use `CountDownLatch` when one thread needs to wait for other threads to finish their work.
  Use `CyclicBarrier` when multiple threads must wait for each other at a common synchronization point and then continue together.

 `CyclicBarrier` automatically resets after all participating threads reach the barrier, so it can be used again for the next phase.
 
 `CyclicBarrier` can be used multiple times. Once all registered threads reach the barrier, 
  it automatically resets and becomes ready for the next synchronization cycle. 
  
 `CountDownLatch` cannot be reused because once its count reaches zero, it stays open permanently.

## Interview Answer

> `Semaphore(2)` allows only two threads to access a shared resource simultaneously.

Semaphores are commonly used for connection pools, ATM systems, printer access, rate limiting, and controlling concurrent access
 to limited resources.

# What is a Deadlock?

A **deadlock** occurs when two or more threads are waiting forever for each other to release locks.


# Why Deadlock Happens?

Four conditions exist:

### 1. Mutual Exclusion

Only one thread can hold a lock.

### 2. Hold and Wait

Thread holds one lock and waits for another.

### 3. No Preemption

Locks cannot be forcibly removed.

### 4. Circular Wait

Threads wait in a cycle.

# Interview Answer (1 Minute)

> A deadlock occurs when two or more threads hold different locks and wait indefinitely for each other's locks. 
For example, Thread-1 acquires lock1 and waits for lock2, while Thread-2 acquires lock2 and waits for lock1. 
The most common way to avoid deadlocks is to maintain a consistent lock ordering across all threads. 
If every thread acquires locks in the same sequence, such as lock1 first and then lock2, circular waiting cannot occur and 
deadlocks are prevented.



Same thread can re-enter the lock.

No deadlock.

That's why it's called:

# Interview Answer

> Deadlock occurs because of circular waiting, where Thread-1 holds Lock-A and waits for Lock-B while Thread-2 holds Lock-B and 
waits for Lock-A. A common prevention technique is maintaining the same lock order. `ReentrantLock` 
helps avoid deadlocks by providing `tryLock()`, timed lock acquisition, and interruptible locking. Using `tryLock()`, 
a thread can attempt to acquire a lock and, if unsuccessful, release its current locks and retry later rather than waiting 
indefinitely. This breaks the circular wait condition and helps avoid deadlocks.

# When Should You Use Class-Level Lock?

Use it when protecting:

* Shared counters
* Application-wide caches

> A class-level lock is obtained when a thread enters a static synchronized method.
 The lock belongs to the Class object, not to any individual instance. 
 Therefore, even if two different objects call two different static synchronized methods, 
 only one thread can execute at a time because both require the same Class lock. 
 Class-level locking is mainly used to protect static data. 
 Problems arise when static variables are protected using instance locks, 
 because different objects have different locks but the static data is shared, 
 which can lead to race conditions. Conversely, using a class-level lock for instance data is safe but may 
 unnecessarily reduce concurrency and performance.

 ## Object-Level Lock (Instance-Level Lock) in Java

When a synchronized **instance method** is called, Java acquires a lock on the **object (instance)**, not on the class.

**Important Rule:**

* One object = One lock
* Different objects = Different locks

So, if two threads are using **different objects**, they can execute synchronized methods simultaneously.

***

Because both methods are synchronized and both need the same object's lock.

Java locks the **object** on which the synchronized method is called.

## What is Synchronization in Java?

**Synchronization** is a mechanism that allows **only one thread at a time** to access a shared resource (critical section).

We use synchronization to prevent **race conditions** and ensure **data consistency** when multiple threads access the same data.


> Synchronization is used to control access to shared resources so that only one thread can execute a critical section at a time,
 preventing data corruption and inconsistent results.

## What is a Race Condition?

A **Race Condition** occurs when **two or more threads access and modify the same shared data at the same time**, 
and the final result depends on which thread finishes first.

In simple words:

> Multiple threads are "racing" to update the same resource, causing unpredictable results.


## Why is it called a Race Condition?

Because threads are competing (racing) to access the same resource.

Whichever thread reaches the shared resource first affects the result.

Example:

Thread-1 → Update Database Record

Thread-2 → Update Same Database Record

The final value depends on which thread updates last.

## How Do We Prevent Race Conditions?

### 1. Synchronization

Only one thread can access the critical section at a time.

### 2. Locks

Examples:

* ReentrantLock
* ReadWriteLock

### 3. Atomic Classes

Examples:

* AtomicInteger
* AtomicLong

### 4. Concurrent Collections

Examples:

* ConcurrentHashMap
* CopyOnWriteArrayList


> A Race Condition occurs when multiple threads simultaneously access and modify the same shared resource, 
and the final result depends on the order in which the threads execute. 
It can lead to inconsistent or incorrect data. Race conditions are typically prevented using synchronization, locks, or atomic classes.

## Runnable State vs Running State

### Runnable State

When you call:

Thread t = new Thread();
t.start();

The thread moves to the **RUNNABLE** state.

**Meaning:**

* The thread is ready to run.
* It is waiting for CPU time.
* It sits in the **Ready Queue (Runnable Pool)**.
* The OS/JVM scheduler has not yet assigned a CPU core to execute it.


Runnable Pool

Thread-1
Thread-2
Thread-3
Thread-4


All these threads are ready, but only one may get the CPU immediately.

***

### Running State

A thread enters the **RUNNING** state when the CPU scheduler selects it and gives it a CPU core.


public void run() {
    System.out.println("Running...");
}


Now instructions inside `run()` are actually executing.


CPU Core
   |
   v
Thread-2 (RUNNING)


The remaining threads stay in the Runnable Pool.

## Interview Answer

> **Runnable State** means the thread is ready and waiting for CPU allocation. It is present in the runnable pool or ready queue. **Running State** means the OS scheduler has selected the thread and the CPU is actively executing its `run()` method. The JVM places threads in the runnable state, but the Operating System scheduler decides when a thread gets CPU time and transitions to the running state based on factors such as CPU availability, scheduling algorithm, and thread priority.


## 3. RUNNING State

The scheduler selects the thread from the runnable pool and executes its `run()` method.


public void run() {
    System.out.println("Thread Running");
}


### Methods Used

* `run()`
* `yield()`

Example:

`yield()` suggests the scheduler give other threads a chance to execute.

Methods:

### B. WAITING

Thread waits indefinitely until another thread wakes it.

Methods:

### C. BLOCKED

Occurs when a thread waits for a lock.

Main thread waits for `t1`.

## Interview Summary (2-Minute Answer)

> Java Thread Life Cycle consists of **NEW → RUNNABLE → RUNNING → WAITING/BLOCKED/TIMED\_WAITING → TERMINATED** states. 
A thread is created in NEW state, moved to RUNNABLE using `start()`, executes in RUNNING state, 
may move to waiting states using methods like `sleep()`, `wait()`, and `join()`, and finally reaches TERMINATED 
when execution completes. Important thread methods include `start()`, `run()`, `sleep()`, `join()`, `yield()`, `interrupt()`,
 and `isAlive()`.

-----------------------------------------------------------multithreading

We use **multithreading** to perform **multiple tasks at the same time** within a single application.

### Why not use a single thread?

		In a single-threaded program, tasks run one after another:

		A Thread represents a separate path of execution within a program. 
		Every Java application has at least one thread called the main thread.

### Why Use Threads?

		Threads are useful when you want to:

		* Run tasks in the background
		* Improve application responsiveness
		* Perform multiple operations simultaneously
		* Utilize multi-core processors efficiently

### Real-Life Example

Think of a web browser:

	* One thread loads the webpage.
	* Another thread downloads images.
	* Another thread responds to user clicks.



Safty with NullPointerException:

Q: What null-safety features are available in Java 17?

Answer: Java 17 does not introduce a new null-safety mechanism. It provides:

	Better NullPointerException messages for easier debugging.
	Pattern matching for instanceof, which avoids explicit null checks and casting.
	Records, where you can enforce null validation in the compact constructor.
	Existing APIs like Optional and Objects.requireNonNull() remain the recommended ways to write null-safe code.
	

1. Helpful NullPointerException Messages

	String name = null;
	System.out.println(name.length());

	Java 17 output:

	Cannot invoke "String.length()" because "name" is null

	For chained calls:

	employee.getDepartment().getManager().getName();

	Output:

	Cannot invoke "Department.getManager()" because
	the return value of "Employee.getDepartment()" is null

	This is one of the biggest improvements for debugging.

2. Pattern Matching for instanceof

		Before:

		if (obj != null && obj instanceof String) {
			String s = (String) obj;
			System.out.println(s);
		}

		Java 17:

		if (obj instanceof String s) {
			System.out.println(s);
		}

		No explicit null check is required because:

		if obj is null → instanceof returns false
		if not null and type matches → s is created automatically
		
3. Records

		Instead of:

		public class Employee {
			private String name;

			public Employee(String name) {
				this.name = name;
			}
		}

		Java 17:

		public record Employee(String name) {

			public Employee {
				java.util.Objects.requireNonNull(name);
			}
		}
Records reduce boilerplate and let you centralize null validation in the compact constructor.

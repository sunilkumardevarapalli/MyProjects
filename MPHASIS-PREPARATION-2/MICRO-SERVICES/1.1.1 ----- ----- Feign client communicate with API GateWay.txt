Production Approach (Using Service Discovery for the Gateway)

	If your API Gateway is also registered with Eureka, you don't need to hardcode its URL.

	Gateway

			spring:
			  application:
				name: api-gateway

	Order Service Feign Client

		@FeignClient(name = "api-gateway")
		public interface PaymentClient {

			@GetMapping("/payment/{id}")
			String payment(@PathVariable Integer id);

		}

Here, Feign:

		Looks up api-gateway in Eureka.
		Connects to one Gateway instance.
		The Gateway looks up payment-service.
		The Gateway forwards the request to the selected Payment Service instance.

	This avoids hardcoded URLs and supports multiple Gateway instances behind a load balancer
	
Direct Service-to-Service vs Through API Gateway

	| Feature                    | Direct Feign (`payment-service`) | Feign via API Gateway        |
	| -------------------------- | -------------------------------- | ---------------------------- |
	| Feign Target               | `payment-service`                | `api-gateway` or Gateway URL |
	| Eureka Lookup              | Payment Service                  | API Gateway                  |
	| Routing                    | Direct                           | Gateway routes request       |
	| Authentication             | Service-specific                 | Centralized at Gateway       |
	| Rate Limiting              | Per service                      | Centralized                  |
	| Logging                    | Per service                      | Centralized                  |
	| API Versioning             | Harder                           | Easier                       |
	| Best for External Requests | No                               | Yes                          |
	| Best for Internal Calls    | Usually Yes                      | Depends on architecture      |


Interview Note:

	A common follow-up question is:

	"Should one microservice always call another through the API Gateway?"

The answer is: Not always.

	External clients (Web, Mobile, Third-party) should almost always go through the API Gateway for authentication, routing,
	 rate limiting, and observability.
	 
	Internal microservice-to-microservice communication is often done directly using Feign, gRPC, or messaging (Kafka/RabbitMQ) 
	to avoid an extra network hop and reduce latency.

	Some organizations intentionally route all traffic through the Gateway for centralized security, auditing, and policy enforcement. 

	Whether this is appropriate depends on the system's architecture and operational requirements.
